OpenHound for BloodHound Enterprise is deployed as a container and can be run in different environments, such
as:

- **Bare-metal or virtual Linux server** using Docker or Podman (e.g., Ubuntu, RHEL);
- **Cloud-native container service** such as AWS Fargate;
- **Self-managed Kubernetes cluster** on-premises or in the cloud;
- **Managed Kubernetes service** such as Amazon EKS, Google GKE, or Azure AKS.

The configuration of OpenHound can be done through environment variables, configuration files or a combination of both. For more information on configuration, see
the [Configuration](configuration.md) section of the documentation. 

The OpenHound container image can be found on [Docker Hub](https://hub.docker.com/r/specterops/openhound) and includes the collectors for Github, Jamf and Okta by default.

## Deployment process

To deploy OpenHound and run a collector:

1. Provision a server with container (docker/podman) support, a Kubernetes cluster, or cloud-native container service to run OpenHound. Ensure that the
   environment meets the system requirements listed below.
2. [Create an OpenHound collector client](https://bloodhound.specterops.io/collect-data/enterprise-collection/create-collector)
3. Deploy the OpenHound container and set the necessary parameters in the configuration file or via environment
   variables. For more information on configuration, see the generic [Configuration](configuration.md) section of the
   documentation and the specific configuration for each collector in the [Collectors](collectors.md) section.
3. [Run an On Demand scan](https://bloodhound.specterops.io/collect-data/enterprise-collection/on-demand-scan)
   or [Create
   a data collection schedule](https://bloodhound.specterops.io/collect-data/enterprise-collection/collection-schedule)


## Enterprise collector configuration

### Minimum configuration parameters
The following parameters must be set in the `[destination.bloodhoundenterprise]` section of the configuration file or via environment variables to run OpenHound and have it upload data to BloodHound Enterprise.


| Runtime option     | Environment Variable      | Description |
|--------------------|---------------------------|------------------------------|
| `token_key`        | DESTINATION__BLOODHOUNDENTERPRISE__TOKEN_KEY | The API token key for authenticating with BloodHound Enterprise.|
| `token_id`  | DESTINATION__BLOODHOUNDENTERPRISE__TOKEN_ID  | The API token ID for authenticating with BloodHound Enterprise. | 
| `url`     | DESTINATION__BLOODHOUNDENTERPRISE__URL | The URL of the BloodHound Enterprise instance. | 
| `interval`    | DESTINATION__BLOODHOUNDENTERPRISE__INTERVAL | The interval at which OpenHound will check for available jobs. |

### Collector-specific configuration parameters
Each collector may have additional required or optional configuration parameters that are specific to the data source being collected. These parameters can also be set in the configuration file or via environment variables. ## Deployment examples

### Kubernetes
A Helm chart is available for deploying OpenHound in a Kubernetes cluster. The chart can be found in the [OpenHound GitHub repository](https://github.com/specterops/openhound).


### Compose
[Docker compose](https://docs.docker.com/compose/) is a tool to define and run multiple containerized applications. A similar approach can be used with Podman.

To get started, create a `config.toml` and `secrets.toml` file with the necessary configuration parameters on the host machine. Below is an example of a `docker-compose.yml` file to deploy OpenHound using Docker Compose, which mounts the configuration files and sets the necessary `DESTINATION__BLOODHOUNDENTERPRISE__COLLECTOR_NAME` variable to run each individual collector.

```yaml
x-scheduler: &scheduler
  image: openhound:<version>
  command: [ "python", "src/scheduler.py" ]
  restart: unless-stopped
  volumes:
   - ${HOME}/.dlt:/home/nonroot/.dlt

  # These environment variables can also be set inside the ~/.dlt/config.toml file 
  # and act as an example. Configuring openhound can be done using environment variables,
  # configuration files or a combination of both.
  environment: &env
    LOG_LEVEL: INFO
    EXTRACT__WORKERS: 8

services:
  openhound-jamf:
    <<: *scheduler
    environment:
      <<: *env
      DESTINATION__BLOODHOUNDENTERPRISE__COLLECTOR_NAME: jamf

  openhound-okta:
    <<: *scheduler
    environment:
      <<: *env
      DESTINATION__BLOODHOUNDENTERPRISE__COLLECTOR_NAME: okta

  openhound-github:
    <<: *scheduler
    environment:
      <<: *env
      DESTINATION__BLOODHOUNDENTERPRISE__COLLECTOR_NAME: github
```


## Server/container requirements

### Hardware

| Component | Minimum Requirement | Recommended Requirement |
|-----------|---------------------|-------------------------|
| CPU       | 2 cores             | 4 cores or more         |
| Memory    | 4 GB                | 16 GB or more           |
| *Storage  | 50 GB               | 100 GB or more          |

*Note*: The requirements vary depending on the size of the environment being collected/converted. Container storage
persistence is not required. However, it is recommended to have a persistent storage for the logs for troubleshooting.
The logs are stored in the `/.local/share/openhound/logs` directory and can be configured to rotate based on time or
file size.
