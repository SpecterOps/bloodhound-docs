This page provides implementation context for the BloodHound Enterprise and Microsoft Sentinel integration.

## Integration architecture

The integration uses Azure-native components to ingest, normalize, and operationalize BloodHound data:

1. **Azure Function App** polls BloodHound Enterprise API endpoints.
1. **Data Collection Rule (DCR)** transforms payloads into Sentinel-compatible schema.
1. **Log Analytics custom tables** store normalized records.
1. **Analytics Rules** evaluate those records and generate incidents.
1. **Sentinel dashboards** visualize findings, posture, audits, and Tier Zero exposure.

## Data model (custom tables)

The integration writes to the following custom tables:

| Table | Purpose |
|---|---|
| `BHEAttackPathsData_CL` | Current attack path findings and attributes |
| `BHEAttackPathsTimelineData_CL` | Attack path state/timeline history |
| `BHEAuditLogsData_CL` | Audit/event activity from BloodHound |
| `BHEFindingTrendsData_CL` | Trend-oriented finding summaries |
| `BHEPostureHistoryData_CL` | Posture and historical posture metrics |
| `BHETierZeroAssetsData_CL` | Tier Zero asset inventory and related context |

## Incident generation model

Incidents are generated by Microsoft Sentinel Analytics Rules that query BloodHound custom tables.

- Rule templates are deployed with the main Sentinel template.
- Incidents begin after rules are created/enabled.
- Incident volume and fidelity depend on data scope settings and polling cadence.

## Polling and deduplication behavior

- Data ingestion is polling-based through Azure Function triggers.
- `Lookup Days` controls initial backfill scope.
- Re-polled records are deduplicated using timestamp/state semantics so dashboards and incidents favor the latest event state.

## Multi-instance and environment mapping

- Deploy one Function App per BloodHound Enterprise instance.
- Environment tagging enables multi-environment filtering in dashboards.
- Configure scope at deployment with selected environments and finding types.

## Operational limitations and non-goals

- Connector health should be validated through Azure monitoring and logs.
- Test connectivity and on-demand ingest are not primary workflow actions in this deployment model.
- Azure Marketplace packaging is future context; current deployment follows ARM template and Function App package flow.
