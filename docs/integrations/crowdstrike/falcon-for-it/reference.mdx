This page explains how the CrowdStrike Falcon for IT integration works with BloodHound Enterprise. It covers the collection architecture, configuration inputs, collected data types, and ingestion workflow.

<Note>
  For setup instructions, see [Install and configure the integration](/integrations/crowdstrike/falcon-for-it/configure).
</Note>

## Integration type

The CrowdStrike Falcon for IT integration is an **endpoint collection and ingestion** integration. It uses Falcon-managed agents, Foundry application logic, and scheduled FFIT queries to collect host telemetry and transform that data for BloodHound Enterprise ingestion.

## Use cases

- Collect BloodHound Enterprise endpoint data from Falcon-managed hosts on a schedule
- Scope collection to selected host groups
- Route collected data through CrowdStrike Next-Gen SIEM for automated transformation and ingestion
- Retry incomplete or failed ingestion attempts until targeted hosts report successfully

## Core design

The integration has six main parts:

| Component | Purpose |
| --- | --- |
| **Falcon agents** | Run on the target endpoints and provide the host execution layer for Falcon for IT queries |
| **Falcon for IT scheduled queries** | Collect sessions, local groups, user rights assignment, and registry data from the selected host groups |
| **Foundry App Repository (LogScale)** | Receives the collected query results as events in CrowdStrike Next-Gen SIEM |
| **Foundry Collections** | Stores state such as job IDs, checkpoints, execution timestamps, and daily status tracking |
| **Foundry function** | Retrieves raw events, transforms them to the BloodHound Enterprise schema, and sends them to the BloodHound Enterprise API |
| **BloodHound Enterprise API** | Accepts the transformed data for final ingestion and analysis |

## Platform assumptions

The source design assumes the following conditions are already true:

- Falcon agents are installed on the target hosts
- The Falcon for IT module is active in the customer environment
- The application can store BloodHound Enterprise connection values in the Foundry Secret Store

## Configuration inputs

Use the following configuration values to control connection, scope, and scheduling behavior.

| Field | Description |
| --- | --- |
| **BloodHound Enterprise URL** | The BloodHound Enterprise tenant that receives the ingested data |
| **BHE Client ID** | The client ID used for BloodHound Enterprise API authentication |
| **BHE Client Secret** | The client secret used for BloodHound Enterprise API authentication |
| **Target Host Groups** | The CrowdStrike host groups that define which assets run the scheduled queries |
| **Collection Frequency** | The schedule interval for automated data collection |
| **Max Retries and Delay** | Optional controls that manage persistent retries when ingestion or egress does not complete successfully |

## Collected data

The packaged FFIT queries collect the following data types for BloodHound Enterprise:

| Data type | Purpose |
| --- | --- |
| **Sessions** | Captures logon session data from the selected hosts |
| **Local groups** | Captures local group membership information |
| **User rights assignment** | Captures local security authority rights assignment data |
| **Registry data** | Captures registry-based information required by the integration |

## End-to-end workflow

The integration follows this workflow:

1. Verify that Falcon agents are installed on the target hosts and that Falcon for IT is active.
1. Install the BloodHound Enterprise Foundry application from CrowdStrike Marketplace.
1. Configure the BloodHound Enterprise URL and API credentials in the application.
1. Select the target host groups and enable the packaged FFIT scheduled queries.
1. Route the resulting events into the Foundry App Repository in CrowdStrike Next-Gen SIEM.
1. Run the Foundry function to fetch the new events, transform them to the BloodHound Enterprise schema, and ingest them into the BloodHound Enterprise API.

## Stateful tracking and retries

The source design calls for stateful tracking so the application can verify collection health and recover from incomplete runs.

| Mechanism | Purpose |
| --- | --- |
| **Daily Status Table** | Tracks which targeted hosts reported successfully during the current 24-hour cycle |
| **Checkpoint system** | Ensures the Foundry function retrieves only events that have not been processed yet |
| **Job ID and execution timestamp tracking** | Confirms that scheduled queries ran before the application attempts data egress |
| **Persistent retries and reconciliation** | Re-runs failed or incomplete work until host coverage and API ingestion succeed |

## Processing flow

During ingestion, the application performs the following sequence:

1. Read the configured host groups, collection frequency, and BloodHound Enterprise connection values.
1. Query the Foundry App Repository for raw events produced by the scheduled FFIT queries.
1. Apply the transformation logic that maps Falcon-specific JSON data to the BloodHound Enterprise schema.
1. Validate the transformed records before ingestion.
1. Send the normalized payload to the BloodHound Enterprise API.
1. Update the state-tracking data so later runs can resume from the correct checkpoint and identify hosts that still need collection.
