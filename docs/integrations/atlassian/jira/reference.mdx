This page explains how the BloodHound Enterprise Jira integration works. It covers the Atlassian Forge app architecture, authentication model, configuration inputs, scheduler behavior, and BloodHound API dependencies.

<Note>
  For setup instructions, see [Install and configure the integration](/integrations/atlassian/jira/configure).
</Note>

## Integration type

The BloodHound Enterprise Jira integration is a **vulnerability management and remediation tracking** integration. It syncs BloodHound Enterprise attack path findings into Jira so security and operations teams can manage remediation work in Jira Software or Jira Service Management.

## Use cases

- Create Jira issues automatically for BloodHound Enterprise findings
- Route findings into Jira Software or Jira Service Management workflows
- Map BloodHound Enterprise zones to Jira priorities and due dates
- Keep Jira aligned with BloodHound Enterprise by closing remediated findings automatically

## Core design

The integration has four main parts:

| Component | Purpose |
| --- | --- |
| **Forge configuration UI** | Stores the BloodHound Enterprise server URL, token ID, token key, sync scope, priority mapping, and auto-closure settings in the project settings page |
| **Sync scheduler** | Pulls findings from BloodHound Enterprise every five minutes and creates or updates Jira issues |
| **Cleanup scheduler** | Runs hourly and closes orphaned Jira issues when the related BloodHound Enterprise finding no longer exists and auto-closure is enabled |
| **Jira project** | Hosts the created issues or incidents and provides the remediation workflow for analysts and operators |

The functional specification describes the following workflow:

1. A Jira administrator installs the Jira Cloud app.
1. A project administrator opens the project settings page and enters the BloodHound Enterprise connection values.
1. The integration tests the connection and unlocks the remaining configuration fields.
1. The sync scheduler polls BloodHound Enterprise and creates or updates Jira issues for matching findings.
1. The cleanup scheduler checks whether previously synced findings still exist and closes the related Jira issues when they no longer do.

## Authentication and secrets

The integration uses BloodHound Enterprise signed requests for API authentication.

Each request includes:

1. `Authorization: bhesignature {token_id}`
1. `RequestDate: {rfc3339_timestamp}`
1. `Signature: {base64_hmac_signature}`

The functional specification states that the app stores the token values in Atlassian Forge encrypted key-value storage and sends requests over HTTPS.

## Configuration inputs

The app exposes the following user-configurable inputs in Jira project settings.

| Input | Required | Description |
| --- | --- | --- |
| **BloodHound Enterprise Domain** | Yes | The URL of your BloodHound Enterprise tenant |
| **Token ID** | Yes | The API token ID used to authenticate requests |
| **Token Key** | Yes | The API token key used to sign requests |
| **Issue Type / Request Type** | Yes | The Jira issue type for Jira Software or the request type for Jira Service Management |
| **BHE Domains** | Yes | The BloodHound Enterprise environments to sync |
| **BHE Zones** | Yes | The BloodHound Enterprise zones to sync |
| **Priority Mapping** | Yes | The mapping between BloodHound Enterprise zones and Jira priorities |
| **Due Days** | Yes | The number of due days to assign for each mapped priority |
| **Enable Auto-Closure** | Yes | Enables or disables automatic closure of orphaned Jira issues |
| **Cleanup Interval** | Yes | The number of days between orphaned-issue closure checks |

## Jira project behavior

The integration adjusts its configuration model based on the Jira project type.

| Capability | Jira Software | Jira Service Management |
| --- | --- | --- |
| **Issue model** | Standard Jira issue type | Incident issue type with a selected request type |
| **Configuration field** | **Issue Type** | **Request Type** |
| **Portal behavior** | Not applicable | Incidents can appear in the JSM portal when the selected request type supports it |

The user guide states that the app detects Jira Service Management projects automatically and filters the request type selector to incident request types.

## Sync model

The source documents describe the following scheduler and processing behavior.

| Setting | Value |
| --- | --- |
| **Sync interval** | Every 5 minutes |
| **Cleanup scheduler** | Every hour |
| **Cleanup interval** | 1 to 10 days, based on configuration |
| **Batch size** | 500 findings per batch |
| **Bulk Jira operations** | 50 issues per API call |
| **Invocation timeout** | 15 minutes |

The integration prevents duplicate issue creation by tracking sync metadata on the Jira issues, including the finding ID and update timestamp.

## Synced Jira issue content

The user guide describes the following content model for created issues.

| Field | Contents |
| --- | --- |
| **Summary** | Zone, domain, affected principal, and finding ID |
| **Description** | Finding title, Graph View link, description, impact details, context, and remediation guidance |
| **Labels** | Zone label and domain identifier |
| **Dates** | Start date and calculated due date |

## API endpoints

The integration depends on the following BloodHound Enterprise API endpoints:

| Endpoint | Purpose |
| --- | --- |
| [`GET /api/v2/available-domains`](/reference/search/get-available-domains) | Lists the environments available to the configured API token |
| [`GET /api/v2/asset-group-tags`](/reference/asset-isolation/get-asset-group-tags) | Lists the available BloodHound Enterprise zones used for sync filtering |
| [`GET /api/v2/domains/{domain_id}/available-types`](/reference/attack-paths/list-available-attack-paths) | Lists the attack path finding types for a selected environment |
| [`GET /api/v2/domains/{domain_id}/details`](/reference/attack-paths/list-domain-attack-paths-details) | Returns detailed attack path records for a selected environment and finding type |
| `GET /api/v2/assets/findings/{finding_type}/title.md` | Returns the title for a finding type |
| `GET /api/v2/assets/findings/{finding_type}/short_description.md` | Returns the short description for a finding type |
| `GET /api/v2/assets/findings/{finding_type}/short_remediation.md` | Returns the short remediation text for a finding type |
| `GET /api/v2/assets/findings/{finding_type}/long_remediation.md` | Returns the long remediation text for a finding type |
| [`GET /api/v2/search`](/reference/search/search-for-objects) | Searches for graph objects by name or object ID |
| [`GET /api/v2/graphs/shortest-path`](/reference/graph/get-the-shortest-path-graph) | Returns the shortest path graph between two objects |
| [`GET /api/v2/azure/{entity_type}`](/reference/azure-entities/get-azure-entity) | Returns Azure entity details for supported object types |

## Error handling

The functional specification calls for clear logging and graceful handling of common API failures.

| Status | Meaning | Expected behavior |
| --- | --- | --- |
| `400` | Bad Request | Log the validation error and stop the current request |
| `401` | Unauthorized | Log the authentication failure and prompt for corrected credentials |
| `403` | Forbidden | Log the authorization failure and stop the current request |
| `404` | Not Found | Log the missing endpoint or resource and skip retry |
| `429` | Too Many Requests | Log the rate-limit condition and retry during a later run |
| `5xx` | Server Error | Log the server-side failure and treat it as a critical sync error |

## Platform dependencies

| Dependency | Role |
| --- | --- |
| **BloodHound Enterprise API** | Source of attack path findings, finding metadata, and object search results |
| **Atlassian Forge** | Hosts the project settings UI, encrypted storage, and scheduled functions |
| **Jira Cloud** | Hosts the destination issues or incidents and provides the workflow used for remediation tracking |
