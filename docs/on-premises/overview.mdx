An on-premises deployment of BloodHound Enterprise is a self-hosted option that runs on infrastructure that you own. It gives you complete control over your deployment while delivering the same powerful capabilities as the SpecterOps-hosted version.

You maintain full control over:

- **Data residency** - All collected data stays within your environment
- **Infrastructure** - Deploy on your own servers or virtual machines
- **Updates** - Control when and how updates are applied
- **Network isolation** - Run in air-gapped or restricted network environments

## SaaS vs on-premises

Both deployment options provide the same core BloodHound Enterprise functionality, but differ in infrastructure management and control.

**Choose on-premises if you:**

- Require data to remain within your infrastructure
- Have regulatory or compliance requirements for data residency
- Need full control over the deployment environment
- Have existing infrastructure and operational expertise
- Prefer to manage updates and maintenance on your own schedule

**Choose SaaS if you:**

- Want SpecterOps to manage infrastructure and updates
- Don't have dedicated infrastructure or Kubernetes expertise
- Want automatic updates and new features as they're released
- Prefer predictable operational costs

## Deployment options

On-premises deployments of BloodHound Enterprise support two options:

### Embedded cluster

This option packages BloodHound Enterprise and a Kubernetes cluster together for deployment on a single Linux host. 

It is based on the open-source Kubernetes distribution [k0s](https://docs.k0sproject.io/stable/), includes a built-in installation UI, runs preflight checks during installation, and updates the cluster and application together. Choose this option if you do not already run Kubernetes.

### Kubernetes Off-The-Shelf (KOTS)

This option installs BloodHound Enterprise into a Kubernetes cluster that you manage by using [KOTS](https://github.com/replicatedhq/kots), which provides a `kubectl` plugin and an in-cluster **Admin Console**.

KOTS provides a UI to configure, install, and update the application in a Kubernetes cluster that you control. Choose this option if you already have mature Kubernetes operations and tooling.

Use the following table to compare deployment and installation options:

| Option | Best fit | Key tradeoffs |
|--------|----------|--------------|
| **Deployment** | | |
| Embedded cluster | No existing Kubernetes operations | Bundled Kubernetes and application update together; less integration with existing K8s tooling |
| KOTS | Mature Kubernetes operations | You manage the cluster lifecycle separately; higher operational complexity |
| **Installation** | | |
| Online | Outbound internet access allowed | Lower install complexity; outbound dependencies required |
| Air gap | No outbound internet access allowed | Higher install complexity; private registry required |
