In the [Deployment Overview](/on-premises/embedded-cluster/overview), we covered the high-level overview of the embedded cluster deployment option for BloodHound Enterprise on-premises. This guide provides step-by-step instructions for installing and configuring BloodHound Enterprise with the embedded cluster installer.

The embedded cluster installer bundles a Kubernetes distribution (k0s) with the BloodHound Enterprise application, so you can deploy everything on a single Linux host without needing to set up Kubernetes separately.

## Pre-install checklist

The following checklist summarizes the install-blocking gates and pre-deployment work that must be completed before running the installer.

<Note>
  For detailed host sizing, collector specifications, supported filesystems and kernel versions, and the complete set of preflight checks the installer enforces, see the full [system requirements](/on-premises/embedded-cluster/system-requirements).
</Note>

<Checklist checklistKey="embedded-cluster-install">
- Provision a dedicated x86-64 Linux host (any distribution with systemd) with root or sudo access, sized per the [BloodHound Enterprise host](/on-premises/embedded-cluster/system-requirements#bloodhound-enterprise-host) specification
- Back the host with SSD storage (P99 write latency ≤10ms) — the installer blocks on spinning disk
- Stand up a [reverse proxy](/on-premises/embedded-cluster/system-requirements#reverse-proxy) with your SSL certificate to terminate HTTPS in front of BloodHound Enterprise
- Create a [DNS A record](/on-premises/embedded-cluster/system-requirements#dns) for the BloodHound Enterprise FQDN that resolves for both users and collectors
- Open inbound TCP `443` (users and collectors) and `30080` (Installation Wizard)
- For online installs, allow outbound HTTPS to `replicated.app`, `proxy.replicated.com`, and `registry.replicated.com`; for air-gapped installs, download the installer bundle ahead of time
</Checklist>

## Installation methods

You can install BloodHound Enterprise using either the **UI-guided installer** or the **CLI-based installer**. Choose the method that fits how you want to configure and deploy the application.

| Method | Use this when |
| --- | --- |
| **Web-based installer** | You want a guided, interactive setup. After you run the install command, the installer prints a URL that you open in a browser (from any machine that can reach the host on the Installation Wizard port) and configures BloodHound Enterprise through the UI. Recommended for first-time installations and one-off deployments. |
| **CLI-based installer** | You want a non-interactive, scriptable install. You provide a pre-built `ConfigValues` file and the Installation Wizard password as flags, and preflight checks run on the command line instead of in the UI. Recommended for automated or repeatable deployments. |

<Tip>
  Both methods produce the same installed cluster and application. The CLI method requires you to prepare a `ConfigValues` file ahead of time; the web-based installer collects those values interactively.
</Tip>

### Web-based installer

The web-based installer provides an interactive setup experience through a web interface. It guides you through configuration steps such as domain setup, certificate management, and database connectivity.

<Steps>
  <Step title="Navigate to the web-based installer">
    The web-based installer is available through the **SpecterOps - BloodHound Enterprise Portal**. You'll work with two interfaces during installation:

    - The **Installer** tab in the portal generates the download and install commands you run on your Linux host. The **Update** tab in the portal tracks active installations.
    - The **Installation Wizard** is a companion UI served from your Linux host on port `30080`. After the initial install completes, you use the Installation Wizard to finish configuration (domain, certificates, database) and run preflight checks.

    If you don't have access to the portal, contact your team administrator or your BloodHound Enterprise account team to request an invitation.

    1. Log in to the [SpecterOps - BloodHound Enterprise Portal](https://enterprise.replicated.com/bloodhound-enterprise).

    1. Click the **Install** tab at the top of the page to open the **Installer page**.

       <Note>
         The installation options displayed on this page depend on the licensing associated with your account.
       </Note>
  </Step>
  <Step title="Choose installation options">

    The first step of the installer prompts you to choose your installation options, such as network availability and whether to show instructions for relocating images to a private registry.

    1. Enter a name for your BloodHound Enterprise instance.

    1. Choose one of the following **Network Availability** options:

       | Option | Use this when |
       | --- | --- |
       | **Outbound requests allowed** | The Kubernetes cluster can make outbound internet requests. |
       | **Outbound requests require HTTPS Proxy** | Outbound traffic must go through a proxy server. |

       <Note>
         If you choose the proxy option, a **Configure proxy URL** field appears on the next page of the installer where you can enter your proxy URL. The installer also adds `--proxy $YOUR_PROXY_URL` to the download _and_ install commands to ensure the installer can access the internet through your proxy.
       </Note>

    1. Click the toggle for **Show instructions to relocate images to your private registry** if your environment requires the cluster to pull container images from a private registry you control.
    
       This is typically required for air-gapped or restricted networks, or to meet internal supply-chain and compliance requirements.

       <Note>
         If you choose this option, the next page of the installer provides a field for entering your private registry URL and instructions for running the necessary commands on your Linux host to:

         - Pull the necessary container images
         - Tag them with your private registry URL
         - Push them to your registry

        The installer also adds `--registry $YOUR_REGISTRY_URL` to the install command to ensure the cluster pulls images from your registry.
       </Note>

       <Frame>
         <img src="/images/on-premises/install-options.png" alt="A screenshot showing the initial installation options in the BloodHound Enterprise web-based installer" />
       </Frame>

    1. Click **Continue**.
  </Step>
  <Step title="Download & install BloodHound Enterprise">
    The portal generates the exact download and install commands for your release, including a signed URL, a short-lived bearer token, and your specific version.
    
    <Note>
      Always copy and run the commands from the portal. The snippets below are placeholder examples that show the shape of each command, not runnable commands.
    </Note>

    <Frame>
      <img src="/images/on-premises/install-instructions.png" alt="A screenshot showing the installation instructions in the BloodHound Enterprise web-based installer" />
    </Frame>

    1. In the **Select Version** field, choose the version of BloodHound Enterprise you want to install.

    1. If you selected the proxy option in the previous installation options, enter your proxy URL in the **Configure proxy URL** field.

       <Note>
         The installer uses this URL to access the internet for downloading container images and other dependencies. The URL must be accessible from the Linux host where you run the installer.
       </Note>

    1. If you selected the private registry option in the previous installation options, enter your registry URL in the provided field.

        <Note>
          The installer uses this URL to pull container images during installation. The URL must be accessible from the Linux host where you run the installer and from the Kubernetes cluster.
        </Note>

    1. SSH into the Linux host where you want to install BloodHound Enterprise.

    1. From the portal, copy and run the command to download the archive of installation assets on your Linux host.

       Placeholder example:

       ```bash
       curl -f "<url>" -H "Authorization: Bearer <token>" -o bloodhound-enterprise-<version>.tar.gz
       ```

    1. From the portal, copy and run the command to extract the archive of installation assets on your Linux host.

       Placeholder example:

       ```bash
       tar -xvzf bloodhound-enterprise-<version>.tar.gz
       ```

       <Note>
         The installer archive contains the embedded cluster application and all necessary dependencies, including the k0s Kubernetes distribution and your license file. The installer will set up everything for you on a single Linux host.
       </Note>

    1. Copy all files from the `assets/` directory to the home directory on your Linux host.

       <Note>
         This is a temporary workaround for a known issue with the installer script.
       </Note>

       Placeholder example:

       ```bash
       cp assets/* ~/
       ```

    1. From the portal, copy and run the install command on your Linux host, providing the path to your license file.

       <Note>
         After extraction, your `license.yaml` file is present alongside all installation assets.
       </Note>

       Placeholder example:

       ```bash
       sudo ./bloodhound-enterprise install --license license.yaml
       ```

    1. Enter an admin password when prompted on the Linux host.
    
       <Note>
         This is the **Installation Wizard password**. You use it to log in to the Installation Wizard later.
       </Note>

    1. When the installer prompts you to generate a self-signed certificate for the Installation Wizard, accept the prompt.

       The installer uses this certificate to serve the Installation Wizard over HTTPS on port `30080`. The install fails if you decline:

       <Note>
         The install command doesn't accept a flag for providing your own certificate at this stage. Generate the self-signed certificate now, then upload your custom certificate later in the Installation Wizard's **Certificates** step.
       </Note>

    1. You can access the Installation Wizard at the following URL after the installation completes:

       ```http
       https://<linux-hostname-or-ip>:30080
       ```

       As mentioned in the pre-installation checklist, the Installation Wizard listens on port `30080`. If you haven't already done so, expose this port on the Linux host.

       <Note>
         Don't click **Finish** on the **Install** tab in the portal yet — see the **Finish** step later in this procedure.
       </Note>
  </Step>
  <Step title="Log in to the Installation Wizard">
    Open the Installation Wizard URL in a browser and enter the admin password that you set when you ran the install command on your Linux host.

    The wizard guides you through the final configuration steps and preflight checks.
  </Step>
  <Step title="Configure">

    The Installation Wizard guides you through the final configuration steps to complete the installation. Follow the prompts to enter the necessary information and complete each step.

    **Domain Configuration**

    Use a lowercase hostname with no underscores. Kubernetes ingress rules require lowercase hostnames.

    - For **production** deployments, this should match the [DNS A record](/on-premises/embedded-cluster/system-requirements#dns) you created for the BloodHound Enterprise FQDN in the pre-install checklist.

    - For **testing** purposes, this can be a hostname that resolves to the Linux host's IP address in your local `/etc/hosts` file. For example:

      ```txt
      192.168.1.50   bloodhound-enterprise.test
      ```

      <Frame>
        <img src="/images/on-premises/installer-domain-config.png" alt="A screenshot showing the domain configuration step of the BloodHound Enterprise web-based installer" />
      </Frame>

      <Note>
        Port `443` must be open on the Linux host and reachable from the browser you use to access the BloodHound Enterprise application.
      </Note>

    **Application Access**

    Configure how users reach the BloodHound Enterprise application.

    <Frame>
      <img src="/images/on-premises/installer-app-access.png" alt="A screenshot showing the application access configuration step of the BloodHound Enterprise web-based installer" />
    </Frame>

    **Certificates**

    Choose whether to have the installer generate a TLS certificate for HTTPS access or upload your own. We recommend uploading your own certificates.

    <Frame>
      <img src="/images/on-premises/installer-certificates.png" alt="A screenshot showing the certificate configuration step of the BloodHound Enterprise web-based installer" />
    </Frame>

    **Database Configuration**

    Choose between the embedded PostgreSQL database or provide connection details for an external PostgreSQL database.

    <Note>
      If you choose to use an external PostgreSQL database, ensure that the database is running PostgreSQL 18 and is reachable from the Linux host where BloodHound Enterprise is installed.
    </Note>

    <Frame>
      <img src="/images/on-premises/installer-database-config.png" alt="A screenshot showing the database configuration step of the BloodHound Enterprise web-based installer" />
    </Frame>
  </Step>
  <Step title="Set Up">
    Configure the network settings the embedded cluster uses to communicate internally and reach external services.
    
    Accept the defaults unless your environment requires specific overrides.

    <Frame>
      <img src="/images/on-premises/installer-setup.png" alt="A screenshot showing the setup step of the BloodHound Enterprise web-based installer" />
    </Frame>
  </Step>
  <Step title="Run Preflight Checks">

    Review and resolve any blocking checks before continuing the installation.

    <Frame>
      <img src="/images/on-premises/installer-preflight-checks.png" alt="A screenshot showing the preflight checks step of the BloodHound Enterprise web-based installer" />
    </Frame>
    
    {/* TODO: Add more details about common preflight check failures and how to resolve them. */}

    When you see the Installation Complete message, click **Finish** to exit the Installation Wizard.

    <Frame>
      <img src="/images/on-premises/installer-success.png" alt="A screenshot showing the installation success step of the BloodHound Enterprise web-based installer" />
    </Frame>
  </Step>
  <Step title="Finish">
    After exiting the Installation Wizard, return to the **Install** tab in the portal and click **Finish**. You'll be redirected to the **Update** tab, where you can view your active installation.

    <Frame>
      <img src="/images/on-premises/online-instances.png" alt="A screenshot showing the Update tab with active installations in the BloodHound Enterprise Portal" />
    </Frame>

    Back on your Linux host, press `Ctrl+C` to exit the installation process and stop the web installer. BloodHound Enterprise continues running and is accessible at the FQDN you configured in the Installation Wizard.
  </Step>
  <Step title="Access BloodHound Enterprise">
    In a browser, navigate to the BloodHound Enterprise FQDN you configured in the Installation Wizard.
    
    <Note>
      The login information is not the same as the Installation Wizard.
    </Note>

    The default username is `admin`. The default password is written to the BloodHound application logs on the Linux host.
    
    To retrieve the password, load the `kubectl` shell environment and inspect the deployment logs:

    1. Load the `kubectl` shell environment:

       ```bash
       sudo ./bloodhound-enterprise shell
       ```

    1. Print the initial admin password from the `bloodhound` deployment's logs:

       ```bash
       kubectl logs -n bloodhound-enterprise deployment/bloodhound | grep "Initial Password Set To:"
       ```

       <Note>
         Targeting the deployment avoids needing to look up the current pod name, which changes between installations and restarts.
       </Note>
  </Step>

#### Ignore preflight checks

In non-production environments where you need to proceed past failing preflight checks (for example, an automated test pipeline running on undersized hardware), you can skip preflight enforcement at install time.

<Warning>
  Skipping host preflight checks is not recommended for production installations. The host preflights guard against conditions — undersized hardware, slow disks, incompatible kernels, missing filesystem options — that cause BloodHound Enterprise to fail or perform poorly under load.
</Warning>

Pass the following flags to the install command:

| Flag | Effect |
| --- | --- |
| `--ignore-host-preflights` | Skips Embedded Cluster host preflight checks (CPU, RAM, disk latency, kernel, filesystem, and so on). |
| `--ignore-app-preflights` | Skips application-level preflight checks defined by BloodHound Enterprise. |
| `--yes` | Auto-confirms the interactive prompt that `--ignore-host-preflights` would otherwise emit. Required for unattended installs. |

Example:

```bash
sudo ./bloodhound-enterprise install \
  --license license.yaml \
  --headless \
  --config-values config-values.yaml \
  --installer-password <password> \
  --ignore-host-preflights \
  --ignore-app-preflights \
  --yes
```

## Troubleshooting

When troubleshooting, you can inspect cluster state and view logs using the `kubectl` command line tool.

<Steps>
  <Step title="SSH into the Linux host">
    SSH into the Linux host where BloodHound Enterprise is installed.
  </Step>
  <Step title="Load the kubectl shell environment">
    Run the following command to load the `kubectl` shell environment:

     ```bash
     sudo ./bloodhound-enterprise shell
     ```

     <Note>
       The appropriate `kubeconfig` is exported, and the location of useful binaries like `kubectl` and Replicated's preflight and support-bundle plugins is added to `PATH`.
     </Note>
  </Step>
  <Step title="Run common troubleshooting commands">
    Use the available binaries as needed. See the following tabs for common cluster inspection commands:

    <Tabs>
      <Tab title="Pods">
      List all pods across every namespace:

      ```bash
      kubectl get pods -A
      ```

      </Tab>
      <Tab title="Services">
      List all services across every namespace:

      ```bash
      kubectl get services -A
      ```

      </Tab>
      <Tab title="Logs">
      View logs for a specific pod:

      ```bash
      kubectl logs -n [namespace] [pod-name]
      ```

      BloodHound application pods run in the `bloodhound-enterprise` namespace.
      </Tab>
    </Tabs>
  </Step>
  <Step title="Exit the kubectl shell">
    When finished, type `exit` or press `Ctrl + D` to exit the shell.
  </Step>
</Steps>
