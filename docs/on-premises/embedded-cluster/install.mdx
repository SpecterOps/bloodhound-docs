In the [Deployment Overview](/on-premises/embedded-cluster/overview), we covered the high-level overview of the embedded cluster deployment option for BloodHound Enterprise on-premises. This guide provides step-by-step instructions for installing and configuring BloodHound Enterprise with the embedded cluster installer.

The embedded cluster installer bundles a Kubernetes distribution (k0s) with the BloodHound Enterprise application, so you can deploy everything on a single Linux host without needing to set up Kubernetes separately.

## Pre-install checklist

The following checklist summarizes the install-blocking gates and pre-deployment work that must be completed before running the installer.

<Note>
  For detailed host sizing, collector specifications, supported filesystems and kernel versions, and the complete set of preflight checks the installer enforces, see the full [system requirements](/on-premises/embedded-cluster/system-requirements).
</Note>

<Checklist checklistKey="embedded-cluster-install">
- Provision a dedicated x86-64 Linux host (any distribution with systemd) with root or sudo access, sized per the [BloodHound Enterprise host](/on-premises/embedded-cluster/system-requirements#bloodhound-enterprise-host) specification
- Back the host with SSD storage (P99 write latency ≤10ms) — the installer blocks on spinning disk
- Stand up a [reverse proxy](/on-premises/embedded-cluster/system-requirements#reverse-proxy) with your SSL certificate to terminate HTTPS in front of BloodHound Enterprise
- Create a [DNS A record](/on-premises/embedded-cluster/system-requirements#dns) for the BloodHound Enterprise FQDN that resolves for both users and collectors
- Open inbound TCP `443` (users and collectors) and `30080` (Admin Console, restricted to admins)
- For online installs, allow outbound HTTPS to `replicated.app`, `proxy.replicated.com`, and `registry.replicated.com`; for air-gapped installs, download the installer bundle ahead of time
</Checklist>

## Installation methods

You can install BloodHound Enterprise using either the **UI-guided installer** or the **CLI-based installer**. Choose the method that fits how you want to configure and deploy the application.

| Method | Use this when |
| --- | --- |
| **Web-based installer** | You want a guided, interactive setup. After you run the install command, the installer prints a URL that you open in a browser (from any machine that can reach the host on the Admin Console port) and configures BloodHound Enterprise through the UI. Recommended for first-time installations and one-off deployments. |
| **CLI-based installer** | You want a non-interactive, scriptable install. You provide a pre-built `ConfigValues` file and the installer password as flags, and preflight checks run on the command line instead of in the UI. Recommended for automated or repeatable deployments. |

<Tip>
  Both methods produce the same installed cluster and application. The CLI method requires you to prepare a `ConfigValues` file ahead of time; the web-based installer collects those values interactively.
</Tip>

#### Ignore preflight checks

In non-production environments where you need to proceed past failing preflight checks (for example, an automated test pipeline running on undersized hardware), you can skip preflight enforcement at install time.

<Warning>
  Skipping host preflight checks is not recommended for production installations. The host preflights guard against conditions — undersized hardware, slow disks, incompatible kernels, missing filesystem options — that cause BloodHound Enterprise to fail or perform poorly under load.
</Warning>

Pass the following flags to the install command:

| Flag | Effect |
| --- | --- |
| `--ignore-host-preflights` | Skips Embedded Cluster host preflight checks (CPU, RAM, disk latency, kernel, filesystem, and so on). |
| `--ignore-app-preflights` | Skips application-level preflight checks defined by BloodHound Enterprise. |
| `--yes` | Auto-confirms the interactive prompt that `--ignore-host-preflights` would otherwise emit. Required for unattended installs. |

Example:

```bash
sudo ./bloodhound-enterprise install \
  --license license.yaml \
  --headless \
  --config-values config-values.yaml \
  --installer-password <password> \
  --ignore-host-preflights \
  --ignore-app-preflights \
  --yes
```

### Filesystem layout

Embedded clusters are built on [k0s](https://k0sproject.io/). All cluster data lives under a single root:

<Tree>
  <Tree.Folder name="/var/lib/embedded-cluster/ <- 40 GB minimum required here" defaultOpen>
    <Tree.File name="bin/ <- EC and k0s binaries" />
    <Tree.Folder name="k0s/ <- k0s data directory" defaultOpen>
      <Tree.Folder name="pki/" defaultOpen>
        <Tree.File name="admin.conf <- cluster kubeconfig" />
      </Tree.Folder>
      <Tree.File name="kubelet/ <- kubelet state and pod volumes" />
      <Tree.File name="containerd/ <- container images and overlay snapshots" />
      <Tree.File name="etcd/ <- etcd data (internal etcd)" />
    </Tree.Folder>
    <Tree.File name="registry/ <- in-cluster registry (airgap installs only)" />
    <Tree.File name="charts/ <- Helm chart cache" />
  </Tree.Folder>
</Tree>

### Relevant paths

The following paths are relevant for troubleshooting the embedded cluster installation:

| Path | Contents |
| --- | --- |
| `/etc/embedded-cluster/` | EC configuration |
| `/var/log/embedded-cluster-install.log` | Installation log |
| `/run/k0s/` | k0s runtime sockets and PID files |
| `/etc/k0s/k0s.yaml` | k0s cluster config (generated, do not edit directly) |
