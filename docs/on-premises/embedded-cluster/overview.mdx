This guide provides a high-level overview of the embedded cluster deployment option, which is the simplest supported way to install BloodHound Enterprise on-premises.

It installs BloodHound Enterprise with a bundled Kubernetes cluster on a single Linux host, so you can focus on Attack Path analysis rather than platform setup.

<Tip>
  If you don't have pre-existing Kubernetes infrastructure or operational expertise, this guide will help you get started quickly.
</Tip>

## Deployment overview

BloodHound Enterprise on-premises has two components:

- **BloodHound Enterprise host (Linux)**
  - Runs the BloodHound Enterprise application
  - Includes a bundled Kubernetes cluster (k0s)
  - Can use an external PostgreSQL database
- **Collector host (Windows)**
  - Runs one or both lightweight collector services
  - SharpHound for on-premises Active Directory and AD CS
  - AzureHound for Entra ID, Azure Resource Manager, and Microsoft Graph

Key data and security characteristics:

- Collectors gather _configuration data_ to map identity relationships
- Data is transmitted over HTTPS with TLS 1.3
- Collectors _do not_ store collected data locally
- You control upload authorization with a [collection schedule](/collect-data/enterprise-collection/collection-schedule) in BloodHound Enterprise

## Installation overview

The installation process involves the following steps:

| Step | What happens | Typical time |
| --- | --- | --- |
| 1. Confirm prerequisites | Validate Linux host, PostgreSQL 16.x, ports, and access. | 0.5-2 hours |
| 2. Install BloodHound Enterprise | Run the guided installer. | 30-60 min |
| 3. Configure connectivity | Point BloodHound Enterprise at PostgreSQL and confirm login works. | 30-60 min |
| 4. Deploy collectors | Install SharpHound and or AzureHound as Windows services. | 5-15 min each |
| 5. Run first collection | Start with the simplest collection level to minimize friction. | Varies |
| 6. Review results | Validate identity Attack Paths and plan next actions. | Varies |
