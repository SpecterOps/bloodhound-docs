title: Switch SharpHound Authentication Methods
---

<img noZoom src="/assets/enterprise-edition-pill-tag.svg" alt="Applies to BloodHound Enterprise only"/>

If you have an existing SharpHound collector client using API Authentication, you can switch to Integrated Windows Authentication (IWA) for enhanced security and simplified credential management.

## Overview

When switching authentication methods for a collector client:

- **Existing authentication tokens are invalidated** - Any previously issued API tokens will no longer be valid for authentication
- **Configuration changes required** - You will need to update both the BloodHound Enterprise tenant configuration and the local SharpHound configuration
- **ADFS configuration is required** - Switching to IWA requires that your ADFS server has already been configured to support IWA clients

## Switching from API Authentication to IWA

### Step 1: Prepare ADFS

Before switching authentication methods, ensure your ADFS server is properly configured to support IWA clients. Follow the [ADFS Configuration guide](/install-data-collector/install-sharphound/configure-adfs-iwa).

### Step 2: Edit the Collector Client

1. In your BloodHound Enterprise tenant, navigate to **Administration** > **Manage Clients**
2. Locate the SharpHound collector client you want to modify
3. Click the hamburger menu and select **Edit Client**
4. Look for the authentication method setting (the exact UI element and placement may vary by release)
5. Change the authentication method from **API Authentication** to **Integrated Windows Authentication (IWA)**
6. Provide the required ADFS configuration details:
   - **Issuer ID (Well-known)**: The ADFS well-known endpoint URL
   - **Issuer Override ID**: Optional override for custom issuer addresses
7. Click **Save** to confirm

### Step 3: Handle Token Invalidation

When you change the authentication method, the system will display a warning that existing API authentication tokens will be invalidated. This is expected behavior. Confirm that you want to proceed.

<Warning>
Any application or script currently using the previous API token will immediately lose its ability to authenticate. Ensure that no other services depend on the old token before switching authentication methods.
</Warning>

### Step 4: Note the New Client Configuration

After switching to IWA, a new **Client Configuration Info** modal will appear containing:

- **Client ID** - Required for ADFS configuration
- **Issuer configuration details** - For reference during setup

Save this information for use in the next step.

### Step 5: Update Local SharpHound Configuration

On the server running SharpHound, update the `settings.json` file with the IWA configuration:

1. Stop the SharpHound Enterprise service: **SharpHound Delegator**
2. Edit `settings.json` located at `%APPDATA%\Roaming\BloodHoundEnterprise\settings.json`
3. Add or update the following fields:
   - `ProviderWellKnown` - The ADFS well-known endpoint URL
   - `ClientId` - The Client ID from the configuration modal
   - `Resource` - The Resource value from the configuration modal
   - `UseIntegratedWindowsAuthForADFS` - Set to `true`
   - `IssuerOverride` - (Optional) Only if you have a custom issuer address
4. Remove or delete `auth.json` - This file is not used when IWA is enabled
5. Save the file
6. Start the SharpHound Enterprise service: **SharpHound Delegator**

### Step 6: Verify the Configuration

1. Check the SharpHound service logs (in `%APPDATA%\Roaming\BloodHoundEnterprise`) to ensure the service started without errors
2. Verify that no authentication errors are logged
3. [Run an on-demand scan](/collect-data/enterprise-collection/on-demand-scan) using the client to confirm authentication is working

## Switching from IWA to API Authentication

To switch from Integrated Windows Authentication back to API Authentication:

1. In your BloodHound Enterprise tenant, navigate to **Administration** > **Manage Clients**
2. Click the hamburger menu on the IWA client and select **Edit Client** (or **Show Client ID** if available)
3. Change the authentication method to **API Authentication**
4. A new **Client Token Info** modal will appear with fresh API credentials
5. Save the token information
6. On the SharpHound server:
   - Stop the SharpHound Enterprise service
   - Replace or update `auth.json` with the new token information
   - Remove the IWA settings from `settings.json` (ProviderWellKnown, ClientId, Resource, etc.)
   - Set `UseIntegratedWindowsAuthForADFS` to `false`
   - Start the SharpHound Enterprise service
7. Verify the configuration by checking logs and running an on-demand scan

## Issuer Address Behavior

### Automatic Trimming

The BloodHound system automatically manages the issuer address from your well-known endpoint:

- **Input**: `https://adfs.example.com/.well-known/openid-configuration`
- **Stored issuer address**: `https://adfs.example.com`

The suffix `/.well-known/openid-configuration` is removed automatically for token validation purposes.

### Issuer Override

If your ADFS configuration uses a different issuer URL for token validation than the base well-known endpoint, use the **Issuer Override ID** field to specify the correct issuer address. This is useful in scenarios where:

- Your ADFS server has multiple issuer identities
- You have a load balancer or proxy that uses a different DNS name for token validation
- Your internal and external ADFS issuer addresses differ

## Troubleshooting Authentication Method Changes

If you encounter issues after switching authentication methods:

1. **Check BloodHound tenant logs** for client authentication failures
2. **Verify ADFS is accessible** from the SharpHound server
3. **Review SharpHound service logs** for detailed error messages
4. **Confirm the Client ID** in ADFS matches the one provided during client creation
5. **Verify claim rules** in ADFS are correctly configured to include the Client ID in the "sub" claim
6. **Check network connectivity** - Ensure firewall rules allow HTTPS traffic to the ADFS server
