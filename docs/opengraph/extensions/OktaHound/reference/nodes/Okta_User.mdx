## Overview

User objects (AKA People) represent individuals who have access to the Okta organization. Each user has a unique identifier, username in the email address format, and various attributes such as email, first name, last name, and status.

In `OktaHound`, users are represented as `Okta_User` nodes.

## Properties

| Name | Source | Type | Description |
| ---- | ------ | ---- | ----------- |
| `id` | `user.Id` | `string` | Unique user identifier. |
| `name` | Constructor argument `login` | `string` | Okta username/login. |
| `displayName` | Constructor argument `displayName` | `string` | User display name. |
| `oktaDomain` | Constructor argument `oktaDomain` | `string` | Okta tenant domain used by the collector. |
| `login` | Constructor argument `login` | `string` | User login/UPN value. |
| `email` | Constructor argument `email` | `string` | Primary email address. |
| `firstName` | Constructor argument `firstName` | `string` | User first/given name. |
| `lastName` | Constructor argument `lastName` | `string` | User last/family name. |
| `title` | Profile attribute `title` | `string` | Job title from user profile when present. |
| `department` | Profile attribute `department` | `string` | Department value from user profile when present. |
| `city` | Profile attribute `city` | `string` | City/location value from user profile when present. |
| `state` | Profile attribute `state` | `string` | State/region value from user profile when present. |
| `countryCode` | Profile attribute `countryCode` | `string` | ISO-like country code from user profile when present. |
| `status` | Constructor argument `status` | `string` | User lifecycle status. |
| `enabled` | Derived from `status` | `bool` | Boolean status projection used by BloodHound. |
| `hasRoleAssignments` | Constructor argument `hasRoleAssignments` | `bool` | Indicates whether the user has role assignments. |
| `credentialProviderName` | `credentials.Provider.Name` | `string` | Authentication provider name for this user. |
| `credentialProviderType` | `credentials.Provider.Type` | `string` | Authentication provider type for this user. |
| `managerId` | Profile manager attribute when present | `string` | Manager identifier from user profile synchronization. |
| `activated` | Constructor argument `activated` | `datetime` | Timestamp when the user account was activated. |
| `created` | Constructor argument `created` | `datetime` | User creation timestamp. |
| `passwordChanged` | Constructor argument `passwordChanged` | `datetime` | Timestamp when the password was last changed. |
| `lastLogin` | Constructor argument `lastLogin` | `datetime` | Timestamp of the most recent successful login. |
| `lastUpdated` | Constructor argument `lastUpdated` | `datetime` | Last profile/update timestamp. |

## User Status

User status can have [multiple values](https://developer.okta.com/docs/api/openapi/okta-management/management/tag/User), as illustrated below:

![Okta user status](https://developer.okta.com/docs/api/images/users/okta-user-status.png)

To simplify analysis in BloodHound, the `OktaHound` collector maps the **Status** attribute to the virtual boolean **Enabled** attribute as follows:

| Okta User Status | Enabled | Explanation                      |
|------------------|---------|----------------------------------|
| ACTIVE           | ✅     | User can authenticate.           |
| PASSWORD_EXPIRED | ✅     | User's password has expired but can still authenticate. |
| LOCKED_OUT       | ✅     | User is locked out but can still authenticate after unlocking. |
| PROVISIONED      | ✅     | User is provisioned but cannot authenticate yet. |
| RECOVERY         | ✅     | User is in recovery mode and cannot authenticate. |
| SUSPENDED        | ❌     | User is suspended and cannot authenticate. |
| STAGED           | ❌     | User is staged and cannot authenticate yet. |
| DEPROVISIONED    | ❌     | User is deprovisioned and cannot authenticate. |

<Warning>
This mapping is a simplification and may not cover all edge cases.
Always refer to the actual **Status** attribute for precise user state information.
</Warning>
## Authentication Factors

Okta supports various authentication factors for multi-factor authentication (MFA),
such as SMS, email, push notifications, and hardware tokens.
In case of mobile and desktop applications, these authentication factors are associated with the [Device](Okta_Device) entities.
Other authentication factors, such as YubiKeys and Google Authenticator, are not represented as separate nodes in BloodHound,
but the number of enrolled factors is stored in the `authenticationFactors` attribute of the `Okta_User` nodes.

## Synchronization with External Directories

Users can be synchronized from external directories such as Active Directory (AD) or LDAP. When synchronized, certain attributes may be mapped from the external directory to the Okta user profile.
