When OpenGraph was introduced in BloodHound v8.0.0, it supported a generic data ingestion model that enabled rapid iteration and flexibility for early OpenGraph projects. This model required data payloads to conform to basic node, edge, and metadata schemas only.

To enable enhanced features and improve consistency across OpenGraph projects, BloodHound now supports an extension-based data ingestion model. This model allows you to register and manage the structures that shape your OpenGraph data in BloodHound. It requires data payloads to conform to an extension schema.

<Note>BloodHound still supports generic data ingestion. For new OpenGraph projects and updates to existing projects, use extension-based ingestion to take advantage of enhanced platform behavior and support.</Note>

## Key concepts

| Concept | Description |
|---|---|
| **Data payload** | The extension-based or generic data generated by an OpenGraph collector that you upload to BloodHound. |
| **Extension-based data** | Data payloads that conform to an extension schema, enabling enhanced features and support in BloodHound. |
| **Extension schema** | A schema that defines the structure and behavior of OpenGraph data for a specific identity provider, cloud service, or other platform, including namespace, custom node and edge definitions, environment identification, and findings. |
| **Generic data** | Data payloads that conform to basic OpenGraph node, edge, and metadata schemas only. |
| **Collector** | A tool that authenticates to a third-party platform and generates a data payload that BloodHound can ingest. |

## Extension schema

Extension schemas include:

- Node and edge definitions
- Environment identification
- Findings and remediation

Extension schemas do **not** include:

- Collectors (download separately from the OpenGraph Library)
- Saved queries (upload separately)
- Complex edges (post-processing)
- Default rules for Privilege Zone

## Before you begin

For non-built-in platforms, complete the following checks before uploading an extension bundle:

<Steps>
	<Step title="Get extension artifacts">
		Obtain the extension bundle and collector from the [OpenGraph Library](/opengraph/library) or your internal delivery workflow.
	</Step>
	<Step title="Review required permissions">
		Review collector permissions in the extension-specific setup documentation.
	</Step>
	<Step title="Confirm role access">
		Confirm your role includes extension management and data upload permissions.
	</Step>
	<Step title="Validate version compatibility">
		Verify the collector version is compatible with the extension bundle version.
	</Step>
</Steps>

## Use extension data (users and analysts)

After an admin uploads an extension bundle:

<Steps>
	<Step title="Upload data payloads">
		Upload extension data payloads using the existing file upload flow.
	</Step>
	<Step title="Select extension environment">
		In **Explore**, **Attack Paths**, and **Posture**, select the extension environment when available.
	</Step>
	<Step title="Import saved queries">
		Import extension saved queries separately for MVP.
	</Step>
</Steps>

## Collector and extension updates

Collectors and extensions are versioned separately but must remain compatible.

- Do not update collectors independently without confirming extension compatibility.
- Update collector and extension bundle together whenever possible.
- In Enterprise, coordinate update cycles with your Technical Account Manager.
