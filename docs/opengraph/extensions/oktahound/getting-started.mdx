The OktaHound extension and collector are available for both BloodHound Community and Enterprise editions. The Enterprise version of OktaHound includes additional features and support, but shares the same core data collection capabilities as the Community version.

How you obtain the OktaHound extension and collector depends on your edition of BloodHound:

- **BloodHound Community** users can download the [latest release](https://github.com/SpecterOps/OktaHound/releases) of OktaHound from GitHub.
- **BloodHound Enterprise** customers must contact their Technical Account Manager to obtain the latest release of the Enterprise version of OktaHound.

After you obtain the OktaHound extension and collector, complete the prerequisites before registering the extension in BloodHound and uploading data.

## Prerequisites

Before you begin, ensure that the following prerequisites are met:

| Prerequisite | Description |
| --- | --- |
| [Create an API service application](/opengraph/extensions/oktahound/okta-app-registration) | Provides the necessary authentication for OktaHound to access your Okta environment |
| [Configure and run the collector](/opengraph/extensions/oktahound/collect-data) | Collects data from your Okta environment and generates JSON files for upload to BloodHound |

## Register the Extension

The OktaHound extension includes a schema that tells BloodHound how to model and analyze data from your Okta organization. You must register the extension before you upload data generated by the OktaHound collector.

On the **OpenGraph Management** page, upload the schema file for your BloodHound edition:

- **BloodHound Community** - `bhce-okta-extension.json`
- **BloodHound Enterprise** - `bhe-okta-extension.json`

OktaHound also includes supporting schemas for related data sources, including SCIM, GitHub, Jamf, 1Password, and Snowflake. These schemas are required for hybrid attack path analysis, even if you do not use every related collector.

- **SCIM** - `bh-scim-extension.json`
- **GitHub** - `bhce-github-extension.json`
- **Jamf** - `bhce-jamfhound-extension.json`
- **1Password** - `bhce-1passhound-extension.json`
- **Snowflake** - `bhce-snowflake-extension.json`

<Note>These supporting schemas are the same for Community and Enterprise versions of OktaHound.</Note>

You can upload these supporting schemas with the main OktaHound schema or later. However, you must upload all of them before you upload the `okta-graph-hybrid.json` file described in the next section.

Hybrid edge data references node kinds across collectors. If any required extension schema is missing, hybrid queries fail.

## Upload Data to BloodHound

After you complete the prerequisites and register the extension, upload the data collected by OktaHound to BloodHound.

The OktaHound collector generates one or more JSON files in the output directory. Each file contains a different subset of data about your Okta environment, such as general configuration data, Active Directory integration data, and hybrid attack path data.

You must upload the generated JSON files to BloodHound in the following order:

1. `okta-graph.json`
1. `okta-graph-ad.json` (optional — includes Active Directory [AD] integration data)
1. `okta-graph-hybrid.json` (optional — includes hybrid AD/Okta attack path data)

<Note>The AD-related files are only generated when OktaHound detects Active Directory integration in your Okta environment.</Note>

## Import Cypher Queries

OktaHound provides [custom Cypher queries](/opengraph/extensions/oktahound/reference/queries) to help you identify attack paths and misconfigurations in your Okta environment. These queries are included in the `custom-queries` directory of the OktaHound extension.

To use these queries, you must first [import](/analyze-data/explore/cypher-search#import-and-export) the `Queries/*.json` files into BloodHound. You can then run the queries on the **Explore** page.

<Tip>OktaHound also provides [specialized queries](/opengraph/extensions/oktahound/reference/privilege-zone-rules) in a separate `PrivilegeZoneRules` directory for creating or updating [Cypher-based](/analyze-data/privilege-zones/rules#cypher) Privilege Zone rules.</Tip>
