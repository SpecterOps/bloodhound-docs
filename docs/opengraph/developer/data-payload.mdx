This page explains the structure and schema requirements for OpenGraph data payloads. It also provides a minimal viable payload example that you can use as a starting point for your own OpenGraph data.

## Before you begin

<a id="version-requirements"></a>

Full OpenGraph support requires a PostgreSQL graph backend and one of the following editions:

- BloodHound Enterprise
- BloodHound Community v8.0.0+ with [PostgreSQL graph backend](/get-started/custom-installation#postgresql)

  <Note>
    While many OpenGraph features may work on a Neo4j graph database, there are functional and performance limitations (see the [OpenGraph FAQ](/opengraph/faq#why-is-it-taking-so-long-to-ingest-opengraph-data)). For full support, migrate to a PostgreSQL graph database backend.
  </Note>

## File requirements

OpenGraph data payloads must be uploaded as a `.json` or `.zip` file. Additional conditions and constraints include:

- Mixed uploads are supported (for example, SharpHound + OpenGraph).
- ZIP archives can contain multiple payload files.
- Nested ZIP files are not supported.
- If you upload a ZIP, every file in the archive must be a valid OpenGraph payload.

## Data payload structure

All OpenGraph extensions use the same data payload format, but the required fields depend on whether you are building a generic or structured graph.

| Goal | Required | Result |
| --- | --- | --- |
| Search OpenGraph data | <ul><li>`graph.nodes`</li><li>`graph.edges`</li></ul> | <Tooltip tip="A graph that uses only the data payload without structured analysis or pathfinding capabilities." cta="Learn more" href="/opengraph/overview#generic-graphs">Generic graph</Tooltip> |
| Enable enhanced analysis<br/>and customization | <ul><li>`metadata`</li><li>`graph.nodes`</li><li>`graph.edges`</li></ul> | <Tooltip tip="A graph that includes structured analysis, pathfinding, and findings based on an installed extension definition schema." cta="Learn more" href="/opengraph/overview#structured-graphs">Structured graph</Tooltip> |

<Note>
  Structured graphs also require an [extension definition schema](/opengraph/developer/extension-definition-schema) in addition to the data payload.
</Note>

At minimum, a generic graph payload must include the following top-level structure:

```json
{
  "graph": {
    "nodes": [],
    "edges": []
  }
}
```

<Note>
  For structured graphs, `metadata` is also required.
</Note>

The `nodes` and `edges` objects must conform to the minimum JSON schemas. BloodHound validates that the JSON is well-formed and that nodes and edges meet these schema requirements, but it does not enforce additional structure or constraints beyond them.

For field-level details, see:

- [OpenGraph Metadata](/opengraph/developer/metadata)
- [OpenGraph Nodes](/opengraph/developer/nodes)
- [OpenGraph Edges](/opengraph/developer/edges)

After ingest completes, you can [search](/analyze-data/explore/search) OpenGraph data.

<Note>
  **Entity Panels**

  Clicking an OpenGraph node or edge in the graph displays an unstructured list of its properties and values.
</Note>

## Troubleshooting

- **Upload fails before schema checks:** Confirm the file is valid JSON and encoded in UTF-8.

- **Upload fails with payload structure errors:** Confirm the payload has top-level `graph` with both `nodes` and `edges` arrays.

- **ZIP upload fails unexpectedly:** Ensure the archive does not contain nested ZIP files.

## Minimal viable data payload

The following is a minimal example payload for a generic graph. You can use it as a starting point for your own OpenGraph data payload. Copy and paste the following example into a new `.json` file or <a href="/assets/opengraph/opengraph-minimal.json" download>download this example file</a>.

<Tip>
  When working with JSON files, use a plain text editor and UTF-8 encoding. Some
  text editors may introduce unexpected, non-standard characters that can cause
  parsing errors. It's always a good idea to validate your JSON with a
  [linter](https://jsonlint.com/) before uploading it to BloodHound.
</Tip>

```json
{
  "graph": {
    "nodes": [
      {
        "id": "123",
        "kinds": ["Person"],
        "properties": {
          "displayname": "bob",
          "property": "a",
          "name": "BOB"
        }
      },
      {
        "id": "234",
        "kinds": ["Person"],
        "properties": {
          "displayname": "alice",
          "property": "b",
          "name": "ALICE"
        }
      }
    ],
    "edges": [
      {
        "kind": "Knows",
        "start": {
          "value": "123",
          "match_by": "id"
        },
        "end": {
          "value": "234",
          "match_by": "id"
        }
      }
    ]
  }
}
```

To test the ingestion in your BloodHound instance, navigate to **Explore** → **Cypher**. Enter the following query and hit `Run`:

```cypher
match p=()-[:Knows]-()
return p
```
