An OpenGraph extension definition schema tells BloodHound how to display, scope, and analyze custom graph data. In this tutorial, you'll build a small sample extension for a fictional environment and add the schema fields that turn a basic payload into a structured graph.

Follow the examples in order. Each section adds one capability to the same sample extension so you can see how the schema and payload work together in BloodHound. 

<Note>
  This tutorial focuses on schema and payload behavior; extension-specific Cypher queries are outside the current scope.
</Note>

By the end of this tutorial, you have a sample extension definition schema and matching payload that:

- Defines an environment to establish analysis scope
- Styles nodes with display kinds
- Identifies principal kinds for pathfinding and metrics
- Makes relationships traversable for attack path analysis
- Adds a Privilege Zone target for finding evaluation
- Defines relationship findings with context and remediation guidance in BloodHound Enterprise

## Define environments

<img
  noZoom
  src="/assets/enterprise-edition-pill-tag.svg"
  alt="Applies to BloodHound Enterprise only"
  style={{ "width":"20%" }}
/>

Use [`environments.environment_kind`](/opengraph/developer/graph-definition#param-environment-kind) to tell BloodHound which node kind represents an environment for organization, filtering, and analysis.

In a custom extension, this behavior comes from the environment node kind you define in the schema and then reference from the payload.

In the UI, this schema definition manifests as environment selection controls. In BloodHound Enterprise, an environment selector appears on the **Attack Paths**, **Posture**, and **Zone Builder** pages.

<Warning>
  Environment selection controls require enabling the findings feature flag in your BloodHound Enterprise tenant. Contact your account team to enable the feature flag.
</Warning>

<Tabs>
  <Tab title="Example definition">
  ```json
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },
      {
        "name": "EX_User",
        "display_name": "Example User",
        "is_display_kind": true,
        "icon": "user",
        "color": "#d33115"
      }
    ],
    "relationship_kinds": [],
    "environments": [
      {
        "environment_kind": "EX_Domain",
        "source_kind": "EX",
        "principal_kinds": ["EX_User"]
      }
    ]
  }
  ```
  </Tab>
  <Tab title="Example payload">
  ```json
  {
    "metadata": {
      "source_kind": "EX"
    },
    "graph": {
      "nodes": [
        {
          "id": "example-domain-1",
          "kinds": ["EX_Domain"],
          "properties": {
            "name": "corp.example.local",
            "displayname": "Corp Example Domain",
            "environmentid": "example-domain-1",
            "collected": true
          }
        },
        {
          "id": "example-user-1",
          "kinds": ["EX_User"],
          "properties": {
            "name": "alice",
            "displayname": "Alice User",
            "environmentid": "example-domain-1"
          }
        }
      ]
    }
  }
  ```
  </Tab>
  <Tab title="BloodHound behavior">
  - The payload's `metadata.source_kind` value registers `EX` as a source kind and applies it to each node in the payload.
  - BloodHound can organize extension data under the environment represented by `EX_Domain` and expose that environment through the relevant environment selector and filters in the UI.
  - Nodes associated with that environment through `properties.environmentid` are evaluated within that scope.
  - In BloodHound Enterprise, the same environment definition also scopes findings and metrics when the other required analysis conditions are met.
  - If the environment node kind is missing from the schema, or if payload nodes do not point to the environment node through `properties.environmentid`, the environment does not appear or scope data as expected.

  </Tab>
</Tabs>

## Define node styling

Use [`is_display_kind`](/opengraph/developer/graph-definition#param-is-display-kind) to decide which kind controls a node's visual identity in the graph.

Use `true` on the kind that should supply the node's icon and color. Use `false` for secondary kinds that help classify the node but should not define its appearance.

This example builds on the environment definition from the previous section by adding a service account display kind and a secondary high-value classification kind. The service account node has two kinds, but only one kind controls how BloodHound renders the node.

<Tabs>
  <Tab title="Example definition">
  ```jsonc
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },
      {
        "name": "EX_User",
        "display_name": "Example User",
        "is_display_kind": true,
        "icon": "user",
        "color": "#d33115"
      },
      { // [!code ++:8]
        "name": "EX_ServiceAccount",
        "display_name": "Example Service Account",
        "description": "Privileged account used to demonstrate display kinds and attack path findings.",
        "is_display_kind": true,
        "icon": "server",
        "color": "#B42318"
      },
      { // [!code ++:6]
        "name": "EX_HighValue",
        "display_name": "Example High Value",
        "description": "Secondary classification kind that should not control node styling.",
        "is_display_kind": false
      }
    ],
    "relationship_kinds": [],
    "environments": [
      {
        "environment_kind": "EX_Domain",
        "source_kind": "EX",
        "principal_kinds": ["EX_User"]
      }
    ]
  }
  ```
  </Tab>
  <Tab title="Example payload">
  ```jsonc
  {
    "metadata": {
      "source_kind": "EX"
    },
    "graph": {
      "nodes": [
        {
          "id": "example-domain-1",
          "kinds": ["EX_Domain"],
          "properties": {
            "name": "corp.example.local",
            "displayname": "Corp Example Domain",
            "environmentid": "example-domain-1",
            "collected": true
          }
        },
        {
          "id": "example-user-1",
          "kinds": ["EX_User"],
          "properties": {
            "name": "alice",
            "displayname": "Alice User",
            "environmentid": "example-domain-1"
          }
        },
        { // [!code ++:12]
          "id": "example-service-account-1",
          "kinds": [
            "EX_ServiceAccount",
            "EX_HighValue"
          ],
          "properties": {
            "name": "sync-service",
            "displayname": "Sync Service",
            "environmentid": "example-domain-1"
          }
        }
      ]
    }
  }
  ```
  </Tab>
  <Tab title="BloodHound behavior">
  - BloodHound uses `EX_ServiceAccount` for the node's icon and color because it is a display kind.
  - `EX_HighValue` still classifies the node, but it does not affect the visual styling because `is_display_kind` is `false`.
  - If a node has multiple display kinds, BloodHound uses the first display kind in the node's `kinds` array.

## Define principals

<img
  noZoom
  src="/assets/enterprise-edition-pill-tag.svg"
  alt="Applies to BloodHound Enterprise only"
  style={{ "width":"20%" }}
/>

Use [`principal_kinds`](/opengraph/developer/graph-definition#param-principal-kinds) to identify which node kinds BloodHound Enterprise should treat as principals inside an environment.

This setting affects analysis scope. If a node kind is not listed in `principal_kinds`, BloodHound Enterprise can still ingest and display those nodes, but BloodHound Enterprise does not treat them as principals for findings and metric calculations in that environment.

This example builds on the previous definition by adding the existing service account node kind to `principal_kinds`.

<Note>
  **Findings and metrics**

  `principal_kinds` does not create findings by itself, and the payload does not include a separate finding object. BloodHound Enterprise derives the finding from the environment definition, the [traversable relationship kind](/opengraph/developer/extension-example#define-relationships), the [Privilege Zone target](/opengraph/developer/extension-example#add-a-privilege-zone-target), the matching [relationship findings](/opengraph/developer/extension-example#define-relationship-findings) definition, and the uploaded graph data.
</Note>

## Define relationships

Use [`relationship_kinds.is_traversable`](/opengraph/developer/graph-definition#param-is-traversable) to control whether edges of a relationship kind are used for pathfinding and Attack Path detection.

When you set `is_traversable` to `true` on a relationship kind, every edge of that kind inherits the same traversability behavior. Use traversable relationships to represent conditions where the starting node can take control of the ending node to a degree that lets an attacker abuse the ending node's outgoing edges.

This example connects Alice, an `EX_User` principal, to the service account from the previous section. It also connects the service account to the domain environment so the example data contains a complete path from Alice to the domain.

The `EX_CanAssumeServiceAccount` relationship kind is traversable because it represents delegated access that lets the starting node assume the ending node's identity. The `EX_ManagesDomain` relationship kind is also traversable because control of the service account gives control of the domain.

<Tabs>
  <Tab title="Example definition">
  ```jsonc
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },
      {
        "name": "EX_User",
        "display_name": "Example User",
        "is_display_kind": true,
        "icon": "user",
        "color": "#d33115"
      },
      {
        "name": "EX_ServiceAccount",
        "display_name": "Example Service Account",
        "description": "Privileged account used to demonstrate display kinds and attack path findings.",
        "is_display_kind": true,
        "icon": "server",
        "color": "#B42318"
      },
      {
        "name": "EX_HighValue",
        "display_name": "Example High Value",
        "description": "Secondary classification kind that should not control node styling.",
        "is_display_kind": false
      }
    ],
    "relationship_kinds": [ // [!code ++:12]
      {
        "name": "EX_CanAssumeServiceAccount",
        "description": "Ability to assume an example service account.",
        "is_traversable": true
      },
      {
        "name": "EX_ManagesDomain",
        "description": "Ability to manage the example domain.",
        "is_traversable": true
      }
    ],
    "environments": [
      {
        "environment_kind": "EX_Domain",
        "source_kind": "EX",
        "principal_kinds": [
          "EX_User",
          "EX_ServiceAccount"
        ]
      }
    ]
  }
  ```
  </Tab>
  <Tab title="Example payload">
  ```jsonc
  {
    "metadata": {
      "source_kind": "EX"
    },
    "graph": {
      "nodes": [
        {
          "id": "example-domain-1",
          "kinds": ["EX_Domain"],
          "properties": {
            "name": "corp.example.local",
            "displayname": "Corp Example Domain",
            "environmentid": "example-domain-1",
            "collected": true
          }
        },
        {
          "id": "example-user-1",
          "kinds": ["EX_User"],
          "properties": {
            "name": "alice",
            "displayname": "Alice User",
            "environmentid": "example-domain-1"
          }
        },
        {
          "id": "example-service-account-1",
          "kinds": [
            "EX_ServiceAccount",
            "EX_HighValue"
          ],
          "properties": {
            "name": "sync-service",
            "displayname": "Sync Service",
            "environmentid": "example-domain-1"
          }
        }
      ],
      "edges": [ // [!code ++:24]
        {
          "kind": "EX_CanAssumeServiceAccount",
          "start": {
            "match_by": "id",
            "value": "example-user-1"
          },
          "end": {
            "match_by": "id",
            "value": "example-service-account-1"
          }
        },
        {
          "kind": "EX_ManagesDomain",
          "start": {
            "match_by": "id",
            "value": "example-service-account-1"
          },
          "end": {
            "match_by": "id",
            "value": "example-domain-1"
          }
        }
      ]
    }
  }
  ```
  </Tab>
  <Tab title="BloodHound behavior">
  - BloodHound includes every `EX_CanAssumeServiceAccount` and `EX_ManagesDomain` edge in pathfinding because both relationship kinds set `is_traversable` to `true`.
  - The path starts at Alice, reaches the service account, and then reaches the domain.
  - This path can contribute to Attack Path detection, findings, and metrics when the domain is in a higher Privilege Zone than Alice.
  - If `is_traversable` is `false`, the relationship can still exist in the graph, but BloodHound does not use edges of that kind for pathfinding, findings, or metrics.
  </Tab>
</Tabs>

## Create a Privilege Zone rule

<img
  noZoom
  src="/assets/enterprise-edition-pill-tag.svg"
  alt="Applies to BloodHound Enterprise only"
  style={{ "width":"20%" }}
/>

Use [Privilege Zones](/analyze-data/privilege-zones/overview) to identify which graph objects represent higher-privilege targets. BloodHound Enterprise generates Attack Path findings when a lower-privileged principal can reach a higher-privileged zone through traversable relationships.

This example builds on the relationship payload from the previous section. The schema and payload define the path from Alice to the domain, but the domain must be in a higher Privilege Zone before BloodHound Enterprise can treat the path as a finding.

<Steps>
  <Step title="Open the pathfinding graph">
    Use the [Pathfinding](/analyze-data/explore/search#pathfinding) tab on the **Explore** page to display the path from `Alice` to `CORP.EXAMPLE.DOMAIN` from the previous example.
  </Step>
  <Step title="Open the domain context menu">
    Right-click the `CORP.EXAMPLE.DOMAIN` node in the graph to open the [context menu](/analyze-data/explore/search#context-menu).
  </Step>
  <Step title="Add the domain to Tier Zero">
    Select **Add to Tier Zero**. BloodHound Enterprise adds the domain to the Tier Zero Privilege Zone and starts analysis.
  </Step>
  <Step title="Wait for analysis">
    Wait for analysis to finish before proceeding to the next step.
  </Step>
</Steps>

After analysis, the example data contains the required privilege boundary:

- Alice is a lower-privileged `EX_User` principal.
- The service account is an `EX_ServiceAccount` principal that manages the domain.
- The domain is a Tier Zero target.

With that boundary in place, the traversable path `alice -> Sync Service -> CORP.EXAMPLE.DOMAIN` can generate a finding when the schema includes a matching `relationship_findings` definition.

## Define relationship findings

<img

Use [`relationship_findings`](/opengraph/developer/graph-definition#relationship_findings) when you want BloodHound Enterprise to attach a finding definition and remediation guidance to a relationship kind that represents an Attack Path condition.

This example builds on the traversable `EX_CanAssumeServiceAccount` relationship and the Tier Zero domain target from the previous sections. The `relationship_findings` entry adds the finding metadata and remediation text that BloodHound Enterprise can display in the UI when matching graph data satisfies the Attack Path conditions.

      {

  - When BloodHound Enterprise detects a qualifying `EX_CanAssumeServiceAccount` edge on a path from Alice to the Tier Zero domain, the finding can display the custom title and remediation content from the schema.
  - The finding is based on the privilege boundary violation: Alice is lower privileged, but she can take control of the service account and reach Tier Zero.
  - If you do not define a matching `relationship_findings` entry, the relationship can still exist in the graph, but BloodHound Enterprise does not have extension-provided finding and remediation text to display for that condition.
  - If the domain is not in a higher Privilege Zone, BloodHound Enterprise does not have the privilege boundary it needs to generate this Attack Path finding.
  </Tab>

## Next steps

Use these examples as a checklist when you design an OpenGraph extension definition schema. Start with the environment model, then add the node kinds, relationship kinds, Privilege Zone targets, and finding definitions that BloodHound needs to display, scope, and analyze your data.

- Review the full [extension definition schema reference](/opengraph/developer/graph-definition) before you define your schema fields.
- Build a matching [graph data payload](/opengraph/developer/graph-data) that uses the node kinds, relationship kinds, environment IDs, and source kind from your schema.
- Check the [node](/opengraph/developer/nodes) and [edge](/opengraph/developer/edges) requirements when you prepare data for ingestion.

## Troubleshoot

If BloodHound does not behave the way you expect, check these common causes first:

- The node or relationship kind name does not exactly match the schema definition.
- The environment node exists, but `properties.collected` is not set to `true`.
- Nodes are missing `properties.environmentid`.
- The environment or payload does not include a registered `source_kind`.
- The relationship kind is not traversable.
- The expected node kind is not included in `principal_kinds`.
- The target node is not in the expected Privilege Zone.
- The finding definition references the wrong `environment_kind` or `relationship_kind`.
