An extension definition schema tells BloodHound how to structure extension-specific [data payloads](/opengraph/developer/graph-data). Use it when you want your extension to produce a <Tooltip tip="A graph generated from a data payload that also conforms to an installed extension definition schema." cta="Learn more" href="/opengraph/overview#structured-graphs">structured graph</Tooltip> and enable advanced analysis features in BloodHound.

<Tip>
  If you only need <Tooltip tip="A graph generated from a data payload that conforms to the minimum JSON schema only." cta="Learn more" href="/opengraph/overview#generic-graphs">generic graph</Tooltip> support, upload a valid OpenGraph data payload and skip this page.
</Tip>

An extension definition schema is a JSON file that OpenGraph extension developers provide for BloodHound users to install on the [OpenGraph Management](/opengraph/extensions/manage) page before they upload data payloads that conform to the schema.

The definitions in the schema are applied to compatible data payloads during ingestion to produce a structured graph.

```json
{
  "schema": { ... },
  "node_kinds": [ ... ],
  "relationship_kinds": [ ... ],
  "environments": [ ... ],
  "relationship_findings": [ ... ]
}
```

<ResponseField name="schema" type="object" required>
  Extension metadata identifying the extension, including version and namespace.
</ResponseField>
<ResponseField name="node_kinds" type="array" required>
  Defines custom node types for extension-specific data payloads and their visual configurations.
</ResponseField>
<ResponseField name="relationship_kinds" type="array" required>
  Defines custom edge types for extension-specific data payloads and their traversability behavior.
</ResponseField>
<ResponseField name="environments" type="array">
  Defines system-level data sources and principal kinds for extension-specific data payloads. Required for findings and metrics in BloodHound Enterprise.
</ResponseField>
<ResponseField name="relationship_findings" type="array">
  Defines findings and remediation guidance that map relationship kinds to analysis output in a target environment.
</ResponseField>

## Constraints

Extension definition schemas must follow the defined JSON schema and satisfy the following constraints.

### Names and namespaces

All `name` fields (except the `schema.name` field) must follow namespacing rules to avoid conflicts with other extensions and built-in graph elements. This doesn't apply to `display_name` fields, which are for human readability only.

<Warning>
  - If names are not unique or properly namespaced, schema uploads fail validation.
  - `TAG` is a reserved namespace prefix for Privilege Zones and cannot be used.
</Warning>

- Must be unique within the extension
- Must be prefixed with your namespace separated by an underscore (for example, `namespace_Name`)
- Must include more than just the namespace

## `schema`

Defines metadata about the extension itself.

```json
{
  "name": "SOOkta",
  "display_name": "Okta Extension (by SpecterOps)",
  "version": "v2.8.1",
  "namespace": "Okta"
}
```

<ResponseField name="name" type="string" required>
  Unique name used by BloodHound to associate ingested data with this schema.
</ResponseField>
<ResponseField name="display_name" type="string" required>
  Human-readable label shown for this extension in BloodHound.
</ResponseField>
<ResponseField name="version" type="string" required>
  Extension schema version in semantic version format, for example `v1.0.0`.
</ResponseField>
<ResponseField name="namespace" type="string" required>
  Prefix used for all extension-defined `node_kinds`, `relationship_kinds`, and `relationship_findings` names.
</ResponseField>


## `node_kinds`

Defines all node types in your graph. Each node kind represents an entity type (for example: user, device, or environment).

```json
[
  {
    "name": "Okta_User",
    "display_name": "Okta User",
    "description": "An Okta user account",
    "is_display_kind": true,
    "icon": "user",
    "color": "#d33115"
  }
]
```

<ResponseField name="name" type="string" required>
  Unique node kind identifier. Must follow [namespacing](/opengraph/developer/graph-definition#namespacing) rules and match the `kinds` field in node data payloads.

  The node schema for an environment must be defined here to be used for findings and metrics in BloodHound Enterprise.
</ResponseField>
<ResponseField name="display_name" type="string" required>
  Human-readable label shown for this node kind in BloodHound.
</ResponseField>
<ResponseField name="description" type="string">
  Optional description that explains what the node kind represents.
</ResponseField>
<ResponseField name="is_display_kind" type="boolean">
  Indicates whether this node kind should be treated as a displayable kind in BloodHound.
</ResponseField>
<ResponseField name="icon" type="string">
  Optional Font Awesome [icon name](https://fontawesome.com/search?s=solid) (without the "fa-" prefix) to show for nodes of this kind in the graph.
</ResponseField>
<ResponseField name="color" type="string">
  Optional Hex color code (in `#RGB` or `#RRGGBB` format, the `#` is required) to apply to nodes of this kind in the graph.
</ResponseField>

## `relationship_kinds`

Defines how nodes can be connected. Each relationship kind represents a specific type of connection between nodes.

```json
[
  {
    "name": "Okta_ResetPassword",
    "description": "Ability to reset passwords or temporary credentials for scoped Okta users",
    "is_traversable": true
  }
]
```

<ResponseField name="name" type="string" required>
  Unique relationship kind identifier. Must follow [namespacing](/opengraph/developer/graph-definition#namespacing) rules and match the `kinds` field in edge data payloads.
</ResponseField>
<ResponseField name="description" type="string">
  Optional description of what the relationship means.
</ResponseField>
<ResponseField name="is_traversable" type="boolean">
  Controls whether edges of this relationship kind are used in pathfinding and analysis.

  When `is_traversable` is set on a relationship kind, all edges of that kind inherit the same traversability behavior. Only <Tooltip tip="Represents a relationship where the starting node can take control of the ending node to a degree that lets an attacker abuse the ending node's outgoing edges" cta="Learn more" href="/resources/edges/traversable-edges">traversable edges</Tooltip> are included in pathfinding and considered for findings and metrics.
</ResponseField>

## `environments`

Defines analysis boundaries and data sources. Each environment represents a system with its own set of <Tooltip tip="A type of object that can authenticate and be assigned permissions within an environment.">principals</Tooltip> and relationships that can be analyzed for findings and metrics in BloodHound Enterprise.

```json
[
  {
    "environment_kind": "Okta_Organization",
    "source_kind": "Okta",
    "principal_kinds": [
      "Okta_User",
      "Okta_Group",
      "Okta_Application",
      "Okta_ApiServiceIntegration"
    ]
  }
]
```

<ResponseField name="environment_kind" type="string" required>
  Node kind that represents the environment boundary for analysis. Must match an environment node kind defined in `node_kinds.name`.

  For findings and metrics in BloodHound Enterprise, the [`graph.nodes.properties.environmentid`](/opengraph/developer/nodes#param-properties-environmentid) fields in the data payload must match this field.
</ResponseField>
<ResponseField name="source_kind" type="string" required>
  Source kind used to associate ingested data with this environment.

  For findings and metrics in BloodHound Enterprise, the [`metadata.source_kind`](/opengraph/developer/metadata#source-kind) field in your data payload must match this field.
</ResponseField>
<ResponseField name="principal_kinds" type="array" required>
  Node kinds that BloodHound should treat as principals within the environment. Must match `node_kinds.name` defined in this schema.
</ResponseField>

## `relationship_findings`

Defines <Tooltip tip="A specific subsection of an Attack Path that BloodHound Enterprise has identified as a high-value remediation point." cta="Learn more" href="/analyze-data/findings/attack-paths#findings">findings</Tooltip> based on relationships. Each finding represents a specific combination of environment and relationship kinds that BloodHound Enterprise should evaluate for high-risk conditions and provide remediation guidance.

Extensions that define findings and remediations work in both Community and Enterprise, but findings are visible in Enterprise only.

<Tip>
  If a finding name would match its associated relationship kind, include `Finding` in the name to distinguish them.
  
  For example:

  - Relationship kind name: `Okta_ResetPassword`
  - Finding name: `Okta_Finding_ResetPassword`
</Tip>

```json
[
  {
    "name": "Okta_Finding_ResetPassword",
    "display_name": "Password Reset Permission Across Privilege Zones",
    "environment_kind": "Okta_Organization",
    "relationship_kind": "Okta_ResetPassword",
    "remediation": {
      "short_description": "Lorem ipsum dolor sit amet, consectetur adipiscing elit.",
      "long_description": "Lorem ipsum dolor sit amet, consectetur adipiscing elit. Sed do eiusmod tempor incididunt ut labore et dolore magna aliqua. Ut enim ad minim veniam, quis nostrud exercitation ullamco laboris nisi ut aliquip ex ea commodo consequat.",
      "short_remediation": "Lorem ipsum dolor sit amet, consectetur adipiscing elit.",
      "long_remediation": "Lorem ipsum dolor sit amet, consectetur adipiscing elit. Sed do eiusmod tempor incididunt ut labore et dolore magna aliqua. Ut enim ad minim veniam, quis nostrud exercitation ullamco laboris nisi ut aliquip ex ea commodo consequat. Duis aute irure dolor in reprehenderit in voluptate velit esse cillum dolore eu fugiat nulla pariatur."
    }
  }
]
```

<ResponseField name="name" type="string" required>
  Unique finding identifier. Must follow [namespacing](/opengraph/developer/graph-definition#namespacing) rules.
</ResponseField>
<ResponseField name="display_name" type="string">
  Human-readable label shown for this finding in BloodHound.
</ResponseField>
<ResponseField name="environment_kind" type="string" required>
  Environment kind where this finding should be evaluated. Must match an `environments.environment_kind` and `node_kinds.name` defined in this schema.
</ResponseField>
<ResponseField name="relationship_kind" type="string" required>
  Relationship kind that triggers this finding when matching edges are present. Must match a `relationship_kinds.name` defined in this schema.
</ResponseField>
<ResponseField name="remediation" type="object">
  Remediation guidance to resolve the finding, including both short and long descriptions.
</ResponseField>
<ResponseField name="remediation.short_description" type="string">
  A concise summary of the recommended remediation action.
</ResponseField>
<ResponseField name="remediation.long_description" type="string">
  A detailed explanation of the remediation, which can include formatting such as markdown for better readability.
</ResponseField>
<ResponseField name="remediation.short_remediation" type="string">
  A concise summary of the steps to remediate the finding.
</ResponseField>
<ResponseField name="remediation.long_remediation" type="string">
  A detailed, step-by-step guide to remediate the finding, which can include formatting such as markdown for better readability.
</ResponseField>

  "node_kinds": [
    {
