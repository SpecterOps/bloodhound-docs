An extension definition schema does more than validate names and kinds. It also controls how BloodHound groups data, renders nodes, and decides which relationships can participate in analysis.

Use this page to understand the practical impact of key schema fields and how they influence BloodHound behavior. Follow the examples in order to build on the same simple domain model and see how to:

- Define environments to establish scope
- Set display kinds to control the visual appearance of nodes
- Define principal kinds to enable finding and metric analysis
- Make relationships traversable to define attack path behavior
- Add relationship findings to provide context and remediation guidance in BloodHound Enterprise

## Define environments

<img
  noZoom
  src="/assets/enterprise-edition-pill-tag.svg"
  alt="Applies to BloodHound Enterprise only"
  style={{ "width":"20%" }}
/>

Use [`environments.environment_kind`](/opengraph/developer/graph-definition#param-environment-kind) to tell BloodHound which node kind represents an environment for organization, filtering, and analysis.

In a custom extension, this behavior comes from the environment node kind you define in the schema and then reference from the payload.

In the UI, this schema definition manifests as environment selection controls. In BloodHound Enterprise, an environment selector appears on the **Attack Paths**, **Posture**, and **Zone Builder** pages.

<Tabs>
  <Tab title="Example definition">
  ```json
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },
      {
        "name": "EX_User",
        "display_name": "Example User",
        "is_display_kind": true,
        "icon": "user",
        "color": "#d33115"
      }
    ],
    "relationship_kinds": [],
    "environments": [
      {
        "environment_kind": "EX_Domain",
        "source_kind": "EX",
        "principal_kinds": ["EX_User"]
      }
    ]
  }
  ```
  </Tab>
  <Tab title="Example payload">
  ```json
  {
    "metadata": {
      "source_kind": "EX"
    },
    "graph": {
      "nodes": [
        {
          "id": "example-domain-1",
          "kinds": ["EX_Domain"],
          "properties": {
            "name": "corp.example.local",
            "displayname": "Corp Example Domain",
            "environmentid": "example-domain-1",
            "collected": true
          }
        },
        {
          "id": "example-user-1",
          "kinds": ["EX_User"],
          "properties": {
            "name": "alice",
            "displayname": "Alice User",
            "environmentid": "example-domain-1"
          }
        }
      ]
    }
  }
  ```
  </Tab>
  <Tab title="BloodHound behavior">
  - BloodHound can organize extension data under the environment represented by `EX_Domain` and expose that environment through the relevant selector or picker controls in the UI.
  - Nodes associated with that environment through `properties.environmentid` are evaluated within that scope.
  - In BloodHound Enterprise, the same environment definition also scopes findings and metrics when the other required analysis conditions are met.
  - If the environment node kind is missing from the schema, or if payload nodes do not point to the environment node through `properties.environmentid`, the environment does not appear or scope data as expected.

  </Tab>
</Tabs>

## Define node styling

Use [`is_display_kind`](/opengraph/developer/graph-definition#param-is-display-kind) to decide which kind controls a node's visual identity in the graph.

Use `true` on the kind that should supply the node's icon and color. Use `false` for secondary kinds that help classify the node but should not define its appearance.

This example builds on the environment definition from the previous section by adding an admin account display kind and a secondary high-value classification kind. The admin account node has two kinds, but only one kind controls how BloodHound renders the node.

<Tabs>
  <Tab title="Example definition">
  ```jsonc
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },
      {
        "name": "EX_User",
        "display_name": "Example User",
        "is_display_kind": true,
        "icon": "user",
        "color": "#d33115"
      },
      { // [!code ++:8]
        "name": "EX_AdminAccount",
        "display_name": "Example Admin Account",
        "description": "Privileged account used to demonstrate display kinds and attack path findings.",
        "is_display_kind": true,
        "icon": "user-shield",
        "color": "#B42318"
      },
      { // [!code ++:6]
        "name": "EX_HighValue",
        "display_name": "Example High Value",
        "description": "Secondary classification kind that should not control node styling.",
        "is_display_kind": false
      }
    ],
    "relationship_kinds": [],
    "environments": [
      {
        "environment_kind": "EX_Domain",
        "source_kind": "EX",
        "principal_kinds": ["EX_User"]
      }
    ]
  }
  ```
  </Tab>
  <Tab title="Example payload">
  ```jsonc
  {
    "metadata": {
      "source_kind": "EX"
    },
    "graph": {
      "nodes": [
        {
          "id": "example-domain-1",
          "kinds": ["EX_Domain"],
          "properties": {
            "name": "corp.example.local",
            "displayname": "Corp Example Domain",
            "environmentid": "example-domain-1",
            "collected": true
          }
        },
        {
          "id": "example-user-1",
          "kinds": ["EX_User"],
          "properties": {
            "name": "alice",
            "displayname": "Alice User",
            "environmentid": "example-domain-1"
          }
        },
        { // [!code ++:12]
          "id": "example-admin-1",
          "kinds": [
            "EX_AdminAccount",
            "EX_HighValue"
          ],
          "properties": {
            "name": "tier-zero-admin",
            "displayname": "Tier Zero Admin",
            "environmentid": "example-domain-1"
          }
        }
      ]
    }
  }
  ```
  </Tab>
  <Tab title="BloodHound behavior">
  - BloodHound uses `EX_AdminAccount` for the node's icon and color because it is a display kind.
  - `EX_HighValue` still classifies the node, but it does not override the visual styling because `is_display_kind` is `false`.
  - If a node has multiple display kinds, BloodHound uses the first display kind in the node's `kinds` array.

## Define principals

<img
  noZoom
  src="/assets/enterprise-edition-pill-tag.svg"
  alt="Applies to BloodHound Enterprise only"
  style={{ "width":"20%" }}
/>

Use [`principal_kinds`](/opengraph/developer/graph-definition#param-principal-kinds) to identify which node kinds BloodHound Enterprise should treat as principals inside an environment.

This setting affects analysis scope. If a node kind is not listed in `principal_kinds`, BloodHound Enterprise can still ingest and display those nodes, but BloodHound Enterprise does not treat them as principals for findings and metric calculations in that environment.

This example builds on the previous definition by adding a service account node kind and including it in `principal_kinds`. The admin account node kind remains defined, but it is not listed as a principal.

<Note>
  **Findings and metrics**

  `principal_kinds` does not create findings by itself, and the payload does not include a separate finding object. BloodHound Enterprise derives the finding from the environment definition, the [traversable relationship kind](/opengraph/developer/schema-behavior-examples#define-relationships), the matching [relationship findings](/opengraph/developer/schema-behavior-examples#define-relationship-findings) definition, and the uploaded graph data.
</Note>

  ```jsonc
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },

        "display_name": "Example User",

      },
      { // [!code ++:8]
        "name": "EX_ServiceAccount",
        "display_name": "Example Service Account",
        "description": "Service account that BloodHound should treat as a principal in the Example environment.",
        "is_display_kind": true,
        "icon": "key",
        "color": "#0F766E"
      },
      {
        "name": "EX_AdminAccount",
        "display_name": "Example Admin Account",
        "description": "Privileged account used to demonstrate display kinds and attack path findings.",
        "is_display_kind": true,
        "icon": "user-shield",
        "color": "#B42318"

      {
        "name": "EX_HighValue",
        "display_name": "Example High Value",
        "description": "Secondary classification kind that should not control node styling.",

      }
    ],

    "environments": [

        "environment_kind": "EX_Domain",

        "principal_kinds": [
          "EX_User",
          "EX_ServiceAccount" // [!code ++]
        ]
      }

  }
  ```
  </Tab>

  - `EX_AdminAccount` is still ingested and displayed in the graph, but BloodHound Enterprise does not evaluate it as a principal for findings and metrics because it is not listed in `principal_kinds`.
  - If a node kind is omitted from `principal_kinds`, BloodHound still ingests and displays nodes of that kind, but it does not evaluate those nodes as principals for findings and metrics.

## Define relationships

Use [`relationship_kinds.is_traversable`](/opengraph/developer/graph-definition#param-is-traversable) to define whether BloodHound can traverse a relationship during graph analysis.

This example connects the `EX_User` principal from the previous section to the `EX_Domain` environment. The relationship can exist in the graph either way, but BloodHound only treats it as an analysis path when the relationship kind is traversable.

<Tabs>
  <Tab title="Example definition">
  ```jsonc
  {
    "schema": {
      "name": "ExampleExtension",
      "display_name": "Example Extension",
      "version": "v1.0.0",
      "namespace": "EX"
    },
    "node_kinds": [
      {
        "name": "EX_Domain",
        "display_name": "Example Domain",
        "is_display_kind": true,
        "icon": "building",
        "color": "#0057b8"
      },
      {
        "name": "EX_User",
        "display_name": "Example User",
        "is_display_kind": true,
        "icon": "user",
        "color": "#d33115"
      },
      {
        "name": "EX_ServiceAccount",
        "display_name": "Example Service Account",
        "description": "Service account that BloodHound should treat as a principal in the Example environment.",
        "is_display_kind": true,
        "icon": "key",
        "color": "#0F766E"
      },
      {
        "name": "EX_AdminAccount",
        "display_name": "Example Admin Account",
        "description": "Privileged account used to demonstrate display kinds and attack path findings.",
        "is_display_kind": true,
        "icon": "user-shield",
        "color": "#B42318"
      },
      {
        "name": "EX_HighValue",
        "display_name": "Example High Value",
        "description": "Secondary classification kind that should not control node styling.",
        "is_display_kind": false
      }
    ],
    "relationship_kinds": [ // [!code ++:7]
      {
        "name": "EX_AdministersDomain",
        "description": "Ability to administer the example domain.",
        "is_traversable": true
      }
    ],
    "environments": [
      {
        "environment_kind": "EX_Domain",
        "source_kind": "EX",
        "principal_kinds": [
          "EX_User",
          "EX_ServiceAccount"
        ]
      }
    ]
  }
  ```
  </Tab>
  <Tab title="Example payload">
  ```jsonc
  {

      "source_kind": "EX"
    },
    "graph": {
      "nodes": [
        {

            "EX_AdminAccount",
            "EX_HighValue"
          ],
          "properties": {
            "name": "tier-zero-admin",
            "displayname": "Tier Zero Admin",
            "environmentid": "example-domain-1"
          }
        },
        {
          "id": "example-service-account-1",
          "kinds": ["EX_ServiceAccount"],
          "properties": {
            "name": "svc-automation",
            "displayname": "Automation Service Account",
            "environmentid": "example-domain-1"
          }
        }
      ],
      "edges": [ // [!code ++:13]
        {
          "kind": "EX_AdministersDomain",
          "start": {
            "match_by": "id",
            "value": "example-user-1"
          },
          "end": {
            "match_by": "id",
            "value": "example-domain-1"
          }
        }
      ]
    }
  }
  ```
  </Tab>
  <Tab title="BloodHound behavior">
  - BloodHound can traverse `EX_AdministersDomain` during analysis because the relationship kind sets `is_traversable` to `true`.
  - The relationship connects the user principal to the domain environment, so it can contribute to Attack Path analysis when the other required analysis conditions are met.
  - If `is_traversable` is `false`, the relationship can still exist in the graph, but BloodHound does not use it as a traversable analysis edge.
  </Tab>
</Tabs>

Use [`relationship_findings`](opengraph/developer/graph-definition#relationship_findings) when you want BloodHound Enterprise to attach a finding definition and remediation guidance to a relationship kind that represents an Attack Path condition.

This example builds on the traversable `EX_AdministersDomain` relationship from the previous section. The `relationship_findings` entry adds the finding metadata and remediation text that BloodHound Enterprise can display when matching graph data satisfies the Attack Path conditions.

## Next steps

Use these examples as a checklist when you design an OpenGraph extension definition schema. Start with the environment model, then add the node kinds, relationship kinds, and finding definitions that BloodHound needs to display, scope, and analyze your data.

- Review the full [extension definition schema reference](/opengraph/developer/graph-definition) before you define your schema fields.
- Build a matching [graph data payload](/opengraph/developer/graph-data) that uses the node kinds, relationship kinds, environment IDs, and source kind from your schema.
- Check the [node](/opengraph/developer/nodes) and [edge](/opengraph/developer/edges) requirements when you prepare data for ingestion.

## Troubleshoot

If BloodHound does not behave the way you expect, check these common causes first:

- The node or relationship kind name does not exactly match the schema definition.
- The environment node exists, but `properties.collected` is not set to `true`.
- Nodes are missing `properties.environmentid`.
- The environment or payload does not include a registered `source_kind`.
- The relationship kind is not traversable.
- The expected node kind is not included in `principal_kinds`.
- The finding definition references the wrong `environment_kind` or `relationship_kind`.
