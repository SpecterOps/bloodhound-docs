The extension definition schema defines the structure of your OpenGraph data.

Use it when you want to produce a **structured graph** with extension-defined behavior.

It tells BloodHound:

- What types of nodes exist
- How those nodes are related
- What environments to analyze
- How findings are generated

If you only need **generic graph** support, upload a valid OpenGraph data payload and skip this page.

<Note>
  Start with [OpenGraph Data](/opengraph/developer/data-payload) for required payload schema guidance. Then return to this page if you want structured graph capabilities.
</Note>

## What this schema adds

An extension definition schema lets extension authors define:

* Node types (`node_kinds`)
* Relationship types (`relationship_kinds`)
* Environments (`environments`)
* Findings (`relationship_findings`)

These definitions are applied to compatible payload data at ingest time.

## Traversability and analysis

Traversability is defined in the extension definition schema, not in the payload edge object.

- Each `relationship_kind` can set `is_traversable`.
- Every ingested edge of that kind inherits that traversability behavior.
- Only traversable edges are used in pathfinding and considered for findings and analysis.

For generic OpenGraph data without an extension definition schema, edge traversability is not defined. Generic data is therefore treated as non-traversable for pathfinding and findings/analysis.

## Schema structure

At a high level:

```json
{
  "schema": { ... },
  "node_kinds": [ ... ],
  "relationship_kinds": [ ... ],
  "environments": [ ... ],
  "relationship_findings": [ ... ]
}
```

<ResponseField name="schema" type="object" required>
  Extension metadata, including identity, display name, and namespace.
</ResponseField>
<ResponseField name="node_kinds" type="object[]" required>
  Node kind definitions that describe the entity types in your structured graph.
</ResponseField>
<ResponseField name="relationship_kinds" type="object[]" required>
  Relationship kind definitions, including traversability behavior.
</ResponseField>
<ResponseField name="environments" type="object[]">
  Environment definitions that map environment nodes, data sources, and principal kinds.
</ResponseField>
<ResponseField name="relationship_findings" type="object[]">
  Finding definitions that map relationship kinds to analysis output in a target environment.
</ResponseField>

## Core sections

### `schema`

Defines metadata about the extension itself.

```json
{
  "name": "Active_Directory",
  "display_name": "Active Directory (SharpHound)",
  "version": "1.0.0",
  "namespace": "AD"
}
```

<ResponseField name="name" type="string" required>
  Unique extension identifier used by BloodHound to register the schema.
</ResponseField>
<ResponseField name="display_name" type="string" required>
  Human-readable extension name shown in product experiences.
</ResponseField>
<ResponseField name="version" type="string" required>
  Extension schema version in semantic version format, for example `1.0.0`.
</ResponseField>
<ResponseField name="namespace" type="string" required>
  Prefix used for all extension-defined kinds and finding names.
</ResponseField>

* `name` must be unique
* `namespace` is used as a prefix for all defined kinds


### `node_kinds`

Defines all node types in your graph.

Each node kind represents an entity (e.g., user, device, environment).

```json
[
  {
    "name": "AD_Domain",
    "display_name": "AD Domain",
    "description": "An Active Directory domain environment",
    "is_display_kind": true
  }
]
```

<ResponseField name="name" type="string" required>
  Unique, namespaced node kind identifier.
</ResponseField>
<ResponseField name="display_name" type="string" required>
  Human-readable label shown for this node kind in BloodHound.
</ResponseField>
<ResponseField name="description" type="string">
  Optional description that explains what the node kind represents.
</ResponseField>
<ResponseField name="is_display_kind" type="boolean">
  Indicates whether this node kind should be treated as a displayable kind in the product.
</ResponseField>

→ See [OpenGraph Nodes](/opengraph/developer/nodes)

### `relationship_kinds`

Defines how nodes can be connected.

Each relationship kind can define traversability through `is_traversable`.

```json
[
  {
    "name": "AD_GenericAll",
    "description": "Full control over target",
    "is_traversable": true
  }
]
```

<ResponseField name="name" type="string" required>
  Unique, namespaced relationship kind identifier.
</ResponseField>
<ResponseField name="description" type="string">
  Optional description of what the relationship means.
</ResponseField>
<ResponseField name="is_traversable" type="boolean">
  Controls whether edges of this relationship kind are used in pathfinding and analysis.
</ResponseField>

→ See [OpenGraph Edges](/opengraph/developer/edges)

### `environments`

Defines analysis boundaries and data sources.

Each environment links:

- An environment node
- A `source_kind`
- The principals within that environment

```json
[
  {
    "environment_kind": "AD_Domain",
    "source_kind": "ADBase",
    "principal_kinds": ["AD_User"]
  }
]
```

<ResponseField name="environment_kind" type="string" required>
  Node kind that represents the environment boundary for analysis.
</ResponseField>
<ResponseField name="source_kind" type="string" required>
  Source kind used to associate ingested data with this environment.
</ResponseField>
<ResponseField name="principal_kinds" type="string[]" required>
  Node kinds that BloodHound should treat as principals within the environment.
</ResponseField>

→ See [OpenGraph Metadata](/opengraph/developer/metadata)

### `relationship_findings`

Defines findings based on relationships.

Findings connect relationships to:

- Environments
- Remediation guidance
- Analysis output

```json
[
  {
    "name": "AD_GenericAll_Finding",
    "environment_kind": "AD_Domain",
    "relationship_kind": "AD_GenericAll"
  }
]
```

<ResponseField name="name" type="string" required>
  Unique, namespaced finding identifier.
</ResponseField>
<ResponseField name="environment_kind" type="string" required>
  Environment kind where this finding should be evaluated.
</ResponseField>
<ResponseField name="relationship_kind" type="string" required>
  Relationship kind that triggers this finding when matching edges are present.
</ResponseField>

<Note>
  Extensions that define findings and remediations work in both Community and Enterprise, but findings are visible in Enterprise only.
</Note>


## Naming and namespacing

All defined names must follow namespacing rules:

- Must be prefixed with your namespace
- Must be unique within the extension
- Must include more than just the namespace

Example:

```
GH_User
GH_Repository
GH_Contains
GH_AdminAccess_Finding
```

<Warning>
  If names are not unique or properly namespaced, the schema upload will fail.
</Warning>

## Validation rules

Some important constraints:

- All `name` fields must be non-empty
- Environment node kinds must also exist in `node_kinds`
- Relationship findings must reference:

  - A valid `relationship_kind`
  - A valid `environment_kind`

<Note>
  Additional validation rules may apply. See [Requirements](/opengraph/developer/data-payload#version-requirements).
</Note>

## Icons and styling

Node kinds can optionally define:

- `icon`
- `color`

Behavior:

- If both are provided → a custom icon is created
- If only one is provided → it is still applied
- Invalid color values (non-hex) will cause errors

→ See [Custom Icons](/opengraph/developer/custom-icons)

## Example schema

```json
{
  "schema": {
    "name": "Active_Directory",
    "display_name": "Active Directory (SharpHound)",
    "version": "1.0.0",
    "namespace": "AD"
  },
  "node_kinds": [
    {
      "name": "AD_Domain",
      "display_name": "AD Domain",
      "description": "An Active Directory domain environment",
      "is_display_kind": true
    }
  ],
  "relationship_kinds": [
    {
      "name": "AD_GenericAll",
      "description": "Full control over target",
      "is_traversable": true
    }
  ],
  "environments": [
    {
      "environment_kind": "AD_Domain",
      "source_kind": "ADBase",
      "principal_kinds": ["AD_User"]
    }
  ],
  "relationship_findings": [
    {
      "name": "AD_GenericAll_Finding",
      "environment_kind": "AD_Domain",
      "relationship_kind": "AD_GenericAll"
    }
  ]
}
```

## How this fits together

To successfully define an extension:

1. Define your node kinds
2. Define relationships between them
3. Declare environments for analysis
4. Map relationships to findings

Each section builds on the others.
